# UI/UX 判断軸 SSOT（ui-ux-principles）

`ui-ux-reviewer` エージェントの判断軸 SSOT。スタック非依存の UI/UX 原則・閾値・3層フレーム・アンチパターンを定義する。

- スタック固有の技術原則（フレーム予算・アセットライフサイクル・レイアウトコスト等）はプロジェクトの `.claude/ui-ux-stack-*.md` に分離し、ui-ux-reviewer が併読する。本ファイルにはスタック固有値を書かない。
- 上流（explorer の調査・planner のプラン）でも UI/UX に関わる設計・改善時に本ファイルを判断軸として参照する。改善案を出す段階で原則が抜けると「遅さを隠すハックの寄せ集め」になるため、入口で原則を効かせる。
- 指摘・改善案には必ず**層タグ（知覚 / 実 / データ設計）**を付す（第2節）。層1だけで層2/3を放置した改善は不完全と判定する。

---

## 1. 応答時間体系（速度の基準）

ユーザーが「速い／遅い」と感じる境界は経験則として確立している。これを基準に応答性を判定する。

### Nielsen の 3 つの応答時間限界

| 限界 | 体感 | 設計含意 |
|---|---|---|
| **0.1 秒（100ms）** | 「瞬時」と感じる | 操作への直接フィードバックはこの範囲。超えると「反応した」と別途示す必要 |
| **1 秒** | 思考が途切れない限界 | この範囲なら待たせても流れを保てる。インジケーター不要だが遅延は感じる |
| **10 秒** | 注意を保てる限界 | これを超える処理は進捗表示 + 完了予測 + 中断手段が必須 |

### RAIL モデル（インタラクション単位の予算）

- **Response（応答）**: 入力に **100ms 以内**でフィードバック。重い処理は即座に「受け付けた」表示を出し、本処理は裏で。
- **Animation（アニメ）**: 各フレーム **16ms** 予算（うち描画に使えるのは実質 ~10ms）。スクロール・ドラッグ・トランジション中にブロッキング処理を挟まない。
- **Idle（アイドル）**: 非緊急処理は **50ms 単位**のチャンクに分割し、入力に即譲る。
- **Load（読み込み）**: 主要コンテンツの体感ロードを抑える（下記 Core Web Vitals で測る）。

### Doherty しきい値

システムとユーザーの応答が互いに **400ms 以内**で続くと、ユーザーは没入し生産性が上がる。逆に 400ms を超える往復は離脱・苛立ちの起点。

### Core Web Vitals（実測の客観指標、2024 INP 導入）

| 指標 | Good | 意味 |
|---|---|---|
| **LCP**（Largest Contentful Paint） | ≤ 2.5s | 主要コンテンツ描画完了 |
| **INP**（Interaction to Next Paint） | ≤ 200ms | 入力→次の描画までの応答性（2024-03 に FID を置換） |
| **CLS**（Cumulative Layout Shift） | ≤ 0.1 | レイアウトの予期せぬズレの累積 |

> 実機計測でしか確定できない体感は「**要実測**」と明記し、推測で Good/Poor を断定しない。

## 2. 3層フレーム（遅さへの対処を分類する）

体感の遅さへの対処を 3 層に分類する。**指摘・改善案には必ずこの層タグを付す**。層1だけの対処は「遅さを隠しただけ」で根本未解決と判定する。

| 層 | 名称 | 内容 | 例 |
|---|---|---|---|
| **層1** | **知覚（perceived）** | 遅さを「隠す／紛らわす」。体感は改善するが処理自体は速くならない | スピナー・スケルトン・フェードイン・プログレスバー・楽観的表示 |
| **層2** | **実（actual）** | 処理そのものを速くする。ブロッキング・無駄な再計算・メインスレッド占有を除去 | 同期処理の非同期化・メモ化・不要再レンダー除去・バンドル削減・遅延ロード |
| **層3** | **データ設計（data）** | 取得戦略を変える。無条件再フェッチをやめキャッシュ前提に | SWR（即表示→裏で再検証）・楽観的更新・プリフェッチ・正規化キャッシュ |

**判定原則**:
- 層1だけで層2/3を放置した改善は**不完全**（FAIL/Medium 相当の指摘対象）。「スピナーを足した」が「毎回サーバーを待っている（層3未対処）」なら根本未解決。
- 理想は層3→層2→層1の順で根本から潰し、層1は「どうしても残る待ち時間」の体感緩和に使う。

## 3. システム状態の可視化とフィードバック（Nielsen #1）

「いま何が起きているか」を常にユーザーに示す。

- **押下フィードバック**: タップ/クリックに 100ms 以内の視覚反応（押下状態・リップル・色変化）。無反応は「効かない」体感を生む。
- **非同期処理の可視化**: 処理中に画面が無反応に見える構造を作らない。ローディング/成功/失敗を必ず可視化する。
- **インジケーターの種別を待ち時間と用途で選ぶ**:
  - **〜1秒**: 原則インジケーター不要（出すなら 1 秒未満で消えるフリッカーに注意）。
  - **スケルトン**: 表示する中身の構造が事前に分かる場合（リスト・カード・記事）。レイアウトを先に確定でき CLS を防ぐ。
  - **スピナー**: 構造が不定・短時間の汎用待ち。
  - **プログレスバー / %表示**: 進捗が定量化でき、かつ長い処理（10 秒級・アップロード等）。完了予測と中断手段を添える。
- **結果の通知**: 成功・失敗を明示。失敗は次節（エラー回復）へ。

## 4. データ取得・状態設計（SWR / 楽観的 UI）

- **無条件再フェッチを避ける**: 画面遷移・再表示のたびにフルフェッチしてスピナーを見せていないか。キャッシュ + **SWR（stale-while-revalidate: 即キャッシュ表示 → 裏で再検証 → 差分だけ更新）**で「即表示」にできないか検討する。
- **楽観的 UI（optimistic update）**: 成功率が高く取り消し可能な操作（いいね・チェック・並べ替え等）は、サーバー応答を待たず即 UI に反映し、失敗時にロールバック。都度サーバー応答を待つのは層3の機会損失。
- **楽観的 UI の誤用に注意**: 誤表示が実害になるデータ（残高・在庫・決済・医療・法務など正確性が安全に直結する値）に楽観的 UI を使わない。ここは確定後表示が正しい。
- **プリフェッチ**: 遷移が高確率で予測できる導線（ホバー・次ページ・ウィザード次ステップ）は先読みして体感ゼロ遷移にできないか。

## 5. アクセシビリティ閾値（WCAG 2.2 AA コア）

ui-ux-reviewer が VERDICT 判定に使う WCAG 2.2 レベル AA のコア閾値。これらの違反は原則 **High**（毎回ユーザーに影響する障壁）。

### コントラスト

| 対象 | 最低比 | 根拠 SC |
|---|---|---|
| 通常テキスト | **4.5:1** | 1.4.3 |
| 大きいテキスト（24px 以上、または 18.66px 以上の太字 ≒ 18pt / 14pt bold） | **3:1** | 1.4.3 |
| UI 部品の境界・状態・グラフィカルオブジェクト（アイコン・グラフの意味を担う線等） | **3:1** | 1.4.11 |

### ターゲットサイズ

| 基準 | 値 | 出典 |
|---|---|---|
| WCAG 2.2 AA 最低（2.5.8、新規） | **24 × 24 CSS px**（間隔で代替可の例外あり） | 2.5.8 |
| 実務推奨 | **44 × 44 CSS px** | 一般推奨 / WCAG 2.5.5 AAA |
| プラットフォーム指針 | iOS **44pt** / Android **48dp** | 各 HIG |

### フォーカス・操作

- **フォーカスの可視化（2.4.7）**: キーボードフォーカスが常に視認できる。`outline:none` で消したまま代替を用意しないのは違反。
- **フォーカスの外観（2.4.11、2.2 新規）**: フォーカスインジケーターが十分な面積とコントラスト（隣接色と 3:1）を持ち、他要素に隠れない。
- **名前・役割・値（4.1.2）**: 全インタラクティブ要素が支援技術にプログラム的に名前・役割・状態を公開する（アイコンのみボタンに aria-label 等）。
- **ドラッグ操作の代替（2.5.7、2.2 新規）**: ドラッグでしかできない操作に単一ポインタ代替を用意。

### 知覚・可読性

- **3 回の閃光（2.3.1）**: 1 秒あたり 3 回を超える閃光を出さない（光感受性発作の回避）。
- **テキストサイズ変更（1.4.4）**: 200% 拡大で内容・機能が失われない。
- **リフロー（1.4.10）**: 320 CSS px 幅相当で横スクロールなしに読める（縦スクロールは可）。
- **テキストの間隔（1.4.12）**: line-height 1.5 / 段落間隔 2× / 文字間隔 0.12em / 単語間隔 0.16em を適用しても破綻しない。

> Web のセマンティック HTML・ARIA・SEO の深い専門は専用 a11y reviewer が別途存在する場合そちらに委ね、ui-ux-reviewer は上記コアに絞る。

## 6. 空・エラー・ローディング状態（Nielsen #5 / #9）

「正常系だけ作って異常系を忘れる」を防ぐ。以下のエッジ状態が設計されているか確認する。

- **空（0 件）**: 「データがない」ことを明示し、次の行動（作成・検索条件変更等）へ導く。空白放置は故障と区別がつかない。
- **エラー（通信失敗・権限・タイムアウト）**: 何が起きたかを平易に伝え、**回復導線（リトライ・戻る・問い合わせ）**を必ず置く。技術スタックを露出する生エラーは出さない。
- **ローディング**: 第3節のインジケーター指針に従う。初回ロードと再検証（SWR）で出し分ける（再検証中は既存内容を消さない）。
- **部分失敗**: 一覧の一部だけ失敗した等、全体を巻き込まずに局所表示する。

## 7. レイアウト・ビジュアル一貫性（Nielsen #4 / #8）

- **一貫性（#4）**: 同種要素は同じ見た目・配置・挙動。デザイントークン（色・間隔・角丸・タイポ）を逸脱した直値（ハードコード）を持ち込まない。プロジェクトの既存トークン体系に従う。
- **最小限の美（#8）**: 情報の階層（サイズ・ウェイト・余白）で主従を示す。装飾過多で本質を埋もれさせない。
- **レイアウトシフトの抑制**: コンテンツ出現時に既存要素を押しのけない。画像・広告・遅延コンテンツは寸法を予約する（CLS ≤ 0.1）。スケルトンは実コンテンツと同寸法にする。
- **整列と余白**: 視線の流れに沿った整列、意味のグルーピングを示す余白（近接の法則）。

## 8. アンチパターン集（これを見たら指摘する）

- **層1偏重**: スピナー/スケルトンで遅さを隠すだけで、層2（ブロッキング除去）・層3（再フェッチ削減）を放置している。
- **無反応な非同期**: 押下後 100ms 以内のフィードバックがなく「効いたか分からない」。
- **無条件再フェッチ**: 画面に戻るたびフルロード + スピナー。SWR/キャッシュで即表示にできるのに毎回待たせる。
- **フリッカー・インジケーター**: 1 秒未満で消えるスピナーが一瞬光って消える（出さない方が良い）。
- **エッジ状態の欠落**: 空・エラー・ロード中の設計がなく、異常時に白画面・無限スピナー・故障同然の見た目になる。
- **回復導線のないエラー**: 失敗を見せるだけでリトライ等の次の手がない。
- **隠しフォーカス**: `outline:none` のまま代替フォーカス表示がなくキーボード操作不能。
- **コントラスト不足の装飾文字**: 薄いグレーのプレースホルダ・キャプションが 4.5:1 を満たさない。
- **過小タップターゲット**: 24px 未満の密集タップ要素（特にモバイル）。
- **レイアウトシフト**: 画像・遅延コンテンツの寸法未予約で読んでいる箇所が飛ぶ。
- **楽観的 UI の誤用**: 残高・在庫・決済など誤表示が実害になる値を確定前に表示。

## 9. 狭幅（モバイル）で顕在化するレイアウト破綻と検出法

PC で正常でもモバイル幅で崩れる典型。§5 のリフロー（320px 横スクロール禁止）・タップ領域を「具体的破綻モード + 修正方向 + 検出法」まで落とす。**幅依存の破綻は狭幅で単独検査する**（PC のスクショだけでは出ない）。

### 破綻パターン（見たら指摘 + 修正方向）

- **コントロールの字潰れ折返し**: ツールバー等の小さなボタン/ナビが横一列に収まらず、container に `flex-wrap` 不在・item に `shrink-0`/`whitespace-nowrap` 不在のため各コントロールが最小幅まで潰され、ラベル（特に CJK は文字間に改行機会があるため）が1文字ずつ縦に折り返す。→ container に `flex-wrap`、item に `shrink-0` + `whitespace-nowrap`。
- **長い無区切り文字列の横溢れ**: URL・長い識別子・ハッシュが `overflow-wrap` 未指定で折り返せず、要素がビューポート幅を突き破りページ全体が横スクロールする。→ 本文リンク/コード/セル等に `overflow-wrap: anywhere`（`break-word`）。flex/grid item は既定 `min-width:auto` で縮まないため祖先に `min-width:0` が要ることがある。
- **見出しの固定スケール過大**: 見出しが幅に応じて縮小されず固定サイズのため、狭幅で長いタイトルが数行に渡り first view を占有する。→ モバイル起点の段階スケール（`clamp()` かブレークポイントで sm/md を小さく、PC 値は据え置き）。
- **`overflow:hidden` による無言の見切れ**: 溢れをスクロールでなく hidden で握りつぶし、コンテンツが黙って切れる（横スクロールの"隠れた双子"。要素の `scrollWidth>clientWidth` かつ overflow hidden/clip かつ ellipsis/line-clamp でない）。→ 溢れ原因を断つ（折返し/レイアウト）か、意図的なら `overflow-x:auto` でスクロール可にし可搬性ヒントを出す。
- **モバイルの極小文字**: 本文級テキストが概ね 11px 未満で可読性を欠く（データ可視化の注記等は例外あり、要判断）。→ モバイルで下限を引き上げる。
- **意図的スクロール要素の可搬性欠如**: wide table 等を `overflow-x:auto` で正しく包んでも、スクロール可能と気づけず最終列が切れて見える。→ スクロール affordance（右端フェード/バー/ヒント）。

### 検出法（重要な落とし穴）

- 横溢れは **`isMobile` エミュレーションを無効にした固定の狭レイアウトビューポート（320 / 360 / 390 CSS px）** で `document.documentElement.scrollWidth > innerWidth` を判定する。**`isMobile:true` はレイアウトビューポートを最広要素に合わせて広げるため `scrollWidth == innerWidth` となり溢れを見逃す**（実測で確認された頻出の誤検出源）。
- 犯人要素は「右端（`rect.right > innerWidth`）を超え、かつ左端は画面内（`rect.left < innerWidth`。画面外の閉じたドロワーを除外）で、`overflow` scroll/auto/clip/hidden の祖先に**封じ込められていない最深の要素**」。祖先で封じ込め済み（意図的スクロール）や `aria-hidden`/`data-state=closed`（オフキャンバス）は除外する。
- タップ領域・折返し・極小文字はまず機械検出（候補列挙）→ **必ず狭幅スクショで視覚確認**（inline リンクや意図的な複数行カード等の false positive を弾く）。
- 対象は**全ページを横断**（横溢れの主因は特定コンテンツ＝長い URL・固有名詞に紐づくため、テンプレのサンプル1枚では見逃す）。
