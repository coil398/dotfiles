# コードベース探索の委譲 — 詳細手順

> このファイルは `~/.claude/CLAUDE.md` 「コードベース探索の委譲」節の詳細。**原則**: 複数ファイルにまたがる調査・パターン探索・広域 grep は自分で直接やらず explorer エージェントに委譲する。自分で直接 Glob/Grep/Read/WebFetch/WebSearch を呼ばず、また Bash の `rg`/`grep`/`find`/`fd`/`ls` による広域探索も行わない。以下は適用範囲・例外・並列運用・字義範囲ガードの詳細。**探索を伴う作業に入る前に Read すること**。

## このルールの適用範囲（「オーケストレーター」とはこれら全てを指す）

- スキル経由で稼働中のメイン Claude（`~/.claude/skills/` 配下の全スキル・プラグイン提供のスキル含む）
- ユーザーから直接依頼を受けたメイン Claude のうち、複数ファイルにまたがる調査・設計相談・レビュー準備など探索が発生する場面
- 「探索ツールが手元にある」「1〜2回だけだから」「軽い確認だから」といった理由で直接呼ぶのは禁止。例外は下記のピンポイント確認のみ

> **サブエージェントの制約**: Claude Code はサブエージェント内からの `Agent` ツール呼び出しを禁止している（`Error: Agent is not available inside subagents.`）。したがってサブエージェント（planner、explorer、implementer、reviewer、tester、retrospector、tech-validator 等）は explorer への委譲ができず、必要に応じて自分の手持ちの `Read`/`Grep`/`Glob` で補助的に確認する。広域探索が必要だと判明した場合は、レポート末尾に「呼び出し元（スキル本体）で別 explorer を起動してほしい」旨を明示して委譲判断を上位に戻すこと。

## 直接読んでよい例外

- 単一ファイルのピンポイント確認（パスが既知・対象は1ファイル・広域パターン探索を兼ねない）は直接 Read してよい
- ユーザーから特定ファイルを提示された場合、その該当ファイルの Read
- git status / git log / git diff など、VCS 由来の軽量な状態確認

## 委譲が必要な典型例（これらは explorer に投げる）

- 「このプロジェクトの○○パターンを調べる」→ Grep/Glob 相当の作業
- 複数ファイルの連続 Read による構造把握
- ディレクトリ階層の列挙、同一レイヤーの既存コンポーネント洗い出し
- ライブラリ公式 doc の WebFetch/WebSearch による裏取り

## 運用ルール

- 初回探索は必須。タスクの規模に関わらず最低1体の explorer を起動する
- 調査対象が独立した複数の領域にまたがる場合は並列起動する。**上限は設けない**。ただし以下の判断軸で領域を分割すること:
  - 各領域が本当に独立か（重複が大きいなら 1 エージェントに統合する）
  - 各領域に十分な調査量があるか（薄い領域は他とまとめる）
  - 領域数が増えすぎたらメインの context が散漫になる感覚を信じる（目安: 4-6 体、超えてもよい）
  - 「3 体までだから」という理由で領域を詰め込まない（過去の上限ルール撤廃）
- 各エージェントには調査対象と期待する出力を具体的に指示する
- 呼び出し元は返ってきた探索レポートをもとに判断・回答・実装を行う
- 初回レポートで不明点があれば追加で explorer を起動する。追加探索は1回に限らず、疑問が解消するまで何度でも繰り返す。推測で埋めない
- explorer への指示には以下の調査観点を含めること:
  - 同一ドメイン・同一レイヤーの既存実装パターン（類似機能がどう実装されているか）
  - 再利用可能な既存ユーティリティ・ヘルパー関数
  - 変更対象の関数内で他分岐が設定しているフィールド・処理の一覧
  - フレームワークが自動処理する機能（新規コードで手動実装すべきでないもの）
- 調査対象がライブラリ・フレームワークの API 仕様や挙動に関わる場合、explorer に対して「公式 README / doc / Issue を WebFetch/WebSearch で裏取りし、参照 URL をレポートに含めること」を明示する。推測や記憶で結論を埋めさせない
- explorer と tech-validator の使い分け:
  - explorer: 既存コードの挙動確認・API 仕様の裏取り・特定バージョンの仕様調査など「調査」が目的
  - tech-validator: ライブラリの採用判断・バージョン選定・同種ライブラリの比較など「選定判断」が目的
  - 調査の結果ライブラリ変更が必要になった場合は tech-validator に引き継ぐ
- explorer の使用モデル等の実装詳細は `~/.claude/agents/explorer.md` で管理する

## 並列 explorer の結果統合

- 複数 explorer のレポートを統合するときは、各レポートの**「対象ファイル + 行範囲」リスト**を正規化して **ファイル単位の和集合** で実装スコープを確定する。特定 explorer の最終提案テーブルだけで実装スコープを決めない
- 同じラベル番号（W-1, W-2 等）が複数 explorer で**違うファイル**を指していることがある。番号で集約せず、ファイル単位で個別に判断する
- 違反すると: ある explorer が catch していた経路を別 explorer のテーブルに集約された情報で上書きしてしまい、修正漏れになる

## explorer 起動プロンプトの字義範囲ガード

- explorer プロンプトに書く調査対象は、**直前会話でユーザーが明示的に扱った具体的トピック**（ファイル名・関数名・機能名・キーワード）にデフォルトで限定する。「網羅的に調べて」「広く確認して」「全体のパターンを」のような広域指示は、本当に網羅が必要なときに限定する
- 直前ターンでユーザーが「これは触らない」「この範囲だけ」と明示したスコープは、explorer プロンプトでも再宣言する。前ターンの限定をスルーした広域探索プロンプトを書かない
- explorer プロンプトを書く前に「ユーザーは直前ターンで具体的に何を扱ったか」を 1〜3 個のキーワードで言語化する。それが思い浮かばないなら、まず人間との対話に戻って範囲を確定する（広域プロンプトを「ひとまず投げる」のは禁止）
- 「最新の動向は？」「全体のパターンを把握して」など字義範囲を超える網羅指示を出す前に、ユーザーが本当にその範囲を望んでいるか自問する。多くの場合、ユーザーは特定のファイル・特定の領域だけを知りたい
- 違反すると: context が散漫化し、「これの話をしてんだよ」「だからないっての」型の叱責につながる
