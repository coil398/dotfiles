## ツール結果の捏造の絶対禁止（最優先・全作業に優先）

ツール（Bash / Read / Edit / Write / Agent / Skill / MCP 等）の**実行結果を自分で書いてはならない**。結果はハーネスが返す実データのみを根拠とする。

- **ツール呼び出しを発行したら、そのアシスタント応答はそこで終える**。`<result>` / `<function_results>` / ツール出力 JSON / サブエージェント・スキルの応答本文を、自分の応答本文に**絶対に書かない**。結果は次ターンでハーネスから受け取る
- 「成功した / 取得した / 保存した / ファイルができた / コンパイル通った」等の**完了主張は、検証ツールの実結果**（`ls` / capture / get_game_object_info / get_unity_logs 等）を受け取ってからのみ行う。発行直後に結果を先取りしない
- **Agent / Skill / 時間のかかる MCP 呼び出しは結果を予測生成しやすい**（"待ち"が発生するため）。これらこそ実結果を待つ。エージェント・スキルの出力を創作しない
- 並列ツール呼び出しでも、各結果は実データを待つ。**まとめて結果を創作しない**
- 本物のツールが成功した流れに乗って、後続呼び出しの成功を検証前に確定テキスト化しない（確証バイアス）。**1 呼び出し = 1 検証**
- **Why**: ツール結果の捏造は、誤った事実に基づく実装ミス・ユーザーの信頼喪失・本物と創作が混ざりデバッグ不能な状態を生む。先例 2026-06-13: `download_assets` の偽「成功（5.2KB 保存）」報告で実ファイル不在（`ls` で 0 件）に気づかず進行し、さらに explorer のコードベース調査結果・git diff・Edit の成功報告まで創作してユーザーに誤報告した

## グローバルファイルの汎用性ルール

グローバルスコープのファイル（`~/.claude/CLAUDE.md`、`~/.claude/agents/`、`~/.claude/skills/`）は **全プロジェクト共通** で使われる。特定プロジェクトに依存する内容を混入させてはならない。

- **禁止**: プロジェクト固有の名前（クラス名、テーブル名、API エンドポイント名、`FooUsecase` のような具体名）、具体的なフレームワーク名やディレクトリパス（`GORM`、`util/` 等）
- **許可**: 汎用的な概念（「usecase」「repository」「controller」等のパターン名）や例示としての仮名（`XxxService` 等）
- 具体的なフレームワーク・ライブラリ・ディレクトリに言及する場合は、プロジェクトの CLAUDE.md またはそこから到達できるファイルに書くこと
- プロジェクト固有の情報はそのプロジェクトの `.claude/CLAUDE.md` またはプロジェクトメモリ（`~/.claude/projects/` 配下）に書くこと
- retrospector がエージェント定義を更新する際もこのルールを厳守すること

## Git ルール

- `git add -A` や `git add .` は **絶対に使わない**。コミット対象のファイルを個別に指定すること
- ファイル単位の `git add <file>` でも、そのファイルの **全 unstaged 変更が stage される** 点に注意。セッション開始時点で既に M 状態のファイルに自分の変更を重ねた場合、自分が意図していない既存差分まで巻き込んでコミットされる。`git commit` の **直前に必ず `git diff --cached` で stage 内容を確認** し、意図しない hunk が混入していれば `git restore --staged <file>` で外すか、`git add -p <file>` で hunk 単位に選別すること
- マージコミットではコンフリクトしたファイルだけを `git add` する
- コミットメッセージに `Co-Authored-By:` 行を付けない
- **`git restore` / `git checkout -- <file>` はユーザーの明示的な指示なしに実行しない**。auto mode でも例外なし。MCP save のノイズ除去・PlayMode のゴミ除去・「多分不要な差分」等を理由に自己判断で restore しない。先例 2026-06-18: Scene YAML を MCP ノイズ除去のつもりで restore → 実質的な座標変更ごと消失。Prefab の fontStyle 変更を「PlayMode のゴミ」と判断して restore → 意図的修正を消失。**restore したい場合は `git diff <file>` の全 hunk をユーザーに提示して判断を仰ぐ**

## ユーザーが実行するコマンドの提示形式

ユーザーがプロンプトに `!` プレフィックスを付けてシェルで実行するコマンドを案内するときは、**コピペ時の崩れに強い形** に正規化してから出すこと。

### 禁則

- **heredoc (`<< EOF ... EOF`) は使わない**。ユーザーの入力環境（プロンプト・エディタの autoindent）が行頭にスペースを差し込み、`EOF` が終端として認識されず破綻する
- 行末バックスラッシュ (`\`) による複数行継続も避ける。同様の理由で改行・インデント混入で壊れる
- 複数行のシェル構文（`for` / `if` / 関数定義 等）も極力避け、必要なら一度ファイルに保存させてから実行する形にする

### 推奨

- ファイル作成は `printf '...\n...\n' > path` か `echo '...' >> path` を **1行で** 提示する（heredoc 不要）
- 連続操作は `&&` で連結した1行コマンドにまとめる
- どうしても複数行が必要なら、Claude 側の Bash ツールで直接実行する選択肢を提示する

### Why

2026-05-20、ユーザーに `tee << 'EOF' ... EOF` 形式の heredoc を `!` 経由で案内した際、コピペ後のターミナル表示で各行頭に 2 スペースが入り `EOF` 終端が効かず、ファイル作成が 2 回失敗した。1 行 `printf` に切り替えて解決した経緯がある。

### How to apply

ユーザーに `!` 付きで打たせるコマンドを書く前に「これは 1 行で書けるか？」を自問する。`<<` / `\` の改行継続を書きそうになったら、`printf` / `&&` 連結に書き直してから提示する。

## 問題の迂回禁止
- 正しい設計に従ったコードが期待通りに動かない場合、勝手に別の手段で迂回しない
- ツールやインフラが使えない場合も、勝手に代替手段で進めない
- 同一のツール呼び出し・コマンドが 2 回続けて失敗したら、3 回目を試さず止める（ブラインドリトライ禁止）。原因が特定でき、かつ再試行で直る合理的根拠があるときのみ再試行してよい
- いずれの場合も原因を報告してユーザーの判断を仰ぐ

## ヘルパー関数の追加禁止
- プロジェクトの既存ユーティリティに同等の関数がないか確認してから書く。その場しのぎのローカルヘルパーを安易に作らない

## スコープ厳守
- 機能実装時は指示されたスコープを厳守する。明示的に依頼されていない隣接モジュールへの変更を勝手に行わない

## 後方互換・dead code の温存禁止
- 「後方互換のため」「将来使うかもしれない」「予約」を理由に **dead code（未使用の prop / 引数 / 関数 / 貫通配線 / フィールド）を勝手に温存しない**。削除を既定とする
- コード中のコメントや reviewer / audit が「後方互換のため残す」と言っていても、その正当化を**鵜呑みにしない**。実際に現行コードで使われているかを確認し、未使用なら削除する（コメントの自己申告は根拠にならない）
- 互換性の維持が本当に必要だと判断したら、温存する前に必ずユーザーに確認する。勝手に互換性を担保しない
- ある機能を廃止したあとに「呼び出し側の互換のため」だけで残っている配線（route → page → component を貫通する未使用 prop 等）は、廃止と同時にチェーンごと除去する

## 自動生成ファイルの扱い
- コード生成ツールの出力ファイルは手動で編集しない。ソース定義（SSOT）のみを編集し、生成コマンドで再生成すること
- どのファイルが自動生成かはプロジェクトの CLAUDE.md を確認する

## API実装ルール
- APIレスポンスの配列フィールドは、要素が0件でも空配列を返す。null/nil を返さない

## Dev server / 長期実行プロセスの扱い

HMR (Hot Module Replacement / Live reload) を備えた dev server（Vite / webpack-dev-server / Next.js dev / `npm run dev` 等）は、**起動したまま使い回す**。毎回 kill / 再起動しない (ユーザー指摘 2026-05-17)。視覚確認 1 回ごとの `pkill` やコミット前の儀式的停止は禁止（HMR を殺すだけで無意味）。例外（`vite.config.*` 等の起動時設定変更・ポート競合・ユーザー明示停止）と原則・禁則の詳細は `~/.claude/dev-server.md` を参照（フロントエンド dev server を起動・操作する作業に入る前に Read）。

## デバッグの進め方
- デバッグ時は、まずユーザーが報告した実際のフロー・パスを確認する。コード分析に入る前に、どのフローが問題になっているか明確にし、不明な場合は質問して確認する
- **「まだ直っていない/再発した」系の報告を調査するときは、報告がどの環境（dev/stg/prod 等）のものかをまず確認する**。環境の明示が無いまま特定環境（特に本番）のデプロイ・リリース状況調査に踏み込まない。先例: ある調査で、ユーザーが一貫して dev 環境の話をしていたのに、本番のデプロイ・リリースタグ状況を勝手に調べに行き、2 度訂正された（「prod の話してねー」「dev だっての」）。環境が不明な発話は「フロー・パスの確認」と同様に質問して確認してから調査に入る
- **うまくいかないときはとにかくログを入れる**。コード分析だけで原因を推測し続けない。`Debug.Log` / `console.log` / `log.Printf` 等をコードに仕込んで実行し、実データで原因を特定する。推測 3 回より実測 1 回の方が速い

## 設計相談時の姿勢
- 命名・URI・型設計などの相談を受けたら、思いつきの単一案を一発で出さない
- まず既存パターンを grep/Read で列挙する（親リソースの意味論、単数形/複数形、接頭辞の有無など）
- 候補を2案以上挙げ、それぞれの既存パターン適合度と逸脱理由を添える
- 直前に自分で調査した事実は必ず設計判断に反映する。調査したのに使わないのは「当てもの」化であり、ユーザーの信頼と時間を浪費する

### 意見聴取と修正アクション要求の見分け
- 「○○って微妙じゃね？」「どう思う？」「○○じゃね？」のような **疑問形・同意確認形** はまず **意見聴取** として扱う。明示的な動詞（直して/修正して/変えて）が無いうちは選択肢の表で煮詰めず、まず自分の見解を述べてユーザーの次の発話を待つ
- 「○○すべき」「これは直す」のような **断定形** で初めて修正前提モードに入る
- 迷ったら「直しますか / 別の使い方にしますか / 気になっただけですか」と一度本人に確認

### 候補提示前のセルフチェック
- ユーザーが過去会話で **既に方針を明示** している論点はないか？ あればその案を選択肢から除外する（言及する場合も「既決方針と矛盾するため除外」と明示）
- 今聞きたい論点と **直交する話題** を選択肢に混ぜていないか？ 直交なら別質問に分けるか、本論決定後に持ち越す
- 候補表に「ユーザー方針との整合性」列を必ず付ける

### 既存構造パターンからの逸脱には事前確認が必要
- 新規ディレクトリ/パッケージ/サブモジュールを切る判断をする前に、同一レイヤー（例: `gateway/`, `repository/`, `service/` 等、外部サービス連携やレイヤー共通のルートディレクトリ）の既存コンポーネントを全件列挙し「単一ファイルか複数ファイル分割か」「サブディレクトリを切るか平置きか」を確認する
- 既存の圧倒的多数が採用している構成から外れる設計（例: 他が全部単一ファイルなのに本件だけサブディレクトリ + 複数ファイル分割）を採用する場合、**実装着手前に必ずユーザーに報告**する
  - 報告内容: 既存 N 件中 M 件の構成 / 今回採用しようとしている構成 / 差分 / 逸脱理由 / 既存パターンに合わせた代替案
- 「将来の拡張性」だけを理由に逸脱構成を採用してはならない。現時点で具体的に複数ファイル分割が必要という根拠がなければ、既存パターンに合わせて単一ファイル構成で始めること
- プロジェクトの CLAUDE.md に「類似機能追加時は既存型を全件確認」等のルールがある場合、それも照合してから設計を確定する

### ユーザー発話の字義範囲解釈

ユーザー発話には「字義通りの解釈」と「推論による拡張解釈」の両方が成立する曖昧な表現が頻出する。メイン Claude は推論側に倒す傾向があり、ユーザーが言ってもいない範囲まで実装・変更・巻き戻しを及ぼす過剰解釈を起こしやすい。**字義範囲を超える解釈が必要な場合は必ず「○○だけ / ○○もですか？」と分けて確認する**。

#### 発火条件（以下のいずれかが発話に含まれるとき）

1. **疑問形・同意確認形**: 「○○じゃね？」「○○すれば良くね？」「○○なくね？」「○○って微妙じゃね？」「どう思う？」など、断定動詞（直して/変えて/消して/作って）を含まない問いかけ
2. **部分否定**: 「○○作る意味ある？」「○○いらなくない？」など、複数の抽象化軸（関数化 / 定数化 / 型エイリアス / モジュール分割 等）のうち**一軸だけを否定**している可能性がある表現
3. **スコープ限定の修正指示**: 「○○だけ直して」「○○の部分だけゼロ値返せ」「○○の名前だけ変えて」など、明示的にスコープを絞った指示
4. **planner 提示済み選択肢からの別案提示**: ユーザーが (A)/(B)/(C) 等の提示済み選択肢から外れた自由記述で「○○すれば良くね？」と返答した場面
5. **実装手段に幅があるとき**: 字義は一意でも、最小実装（定数集約・値の引き回しのみ）と拡張実装（型の全層波及・シグネチャ変更）の選択幅があるとき。発話解釈ではなく実装スコープの選択問題

#### 禁止パターン（字義範囲を超えた拡張解釈の典型）

- **隣接軸への連鎖否定**: 「**メソッド**作る意味ある？」を「定数化もやめろ」と読み替えてリテラル直書きに切り替える等、明示否定されていない別軸まで連鎖的に否定する
- **別案の暗黙拡張**: ユーザーの別案を「複数軸を同時に変える案」と勝手に解釈し、確認なしで複数箇所を書き換える
- **部分否定の全否定化**: 「○○いらなくない？」を「関連する○○も全部いらない」と全否定として処理する
- **スコープ限定指示の関連箇所拡張**: 「○○だけゼロ値返せ」を「required も外す」「関連バリデーションも外す」まで拡張する
- **既存パターン準拠を根拠にした実装スコープ拡張**: 最小実装で字義を満たせるのに「既存がこうだから」だけを理由に全層波及に倒す。形式的対称性よりユーザー指示の最小スコープを優先する

#### Why（なぜこのルールが必要か）

プロジェクト memory に複数の feedback（先例参照）が蓄積されているにもかかわらず、メイン Claude の判断時にこれらが照合されない「memory に書くだけでは矯正できない認知バイアス」が観察されている。memory を増やすだけでは再発が止まらないため、**判断ロジック自体に「字義範囲を超えるなら必ず確認する」ゲートを置く**ことで予防する。

#### How to apply（運用手順）

1. ユーザー発話を受けたら、まず**字義通りの最小解釈**で何を求められているかを特定する
2. その最小解釈で実装に踏み切るのに**情報不足**な場合のみ、考えられる拡張解釈を列挙する
3. 拡張解釈が 1 つ以上ある場合、**実装着手前に必ず**「○○だけですか？それとも ××（拡張範囲）も含みますか？」と分けて確認する
4. 確認後にユーザーが拡張範囲を承認した場合のみ拡張解釈で進める
5. **応答冒頭で動詞 + 目的語を字義抽出して明示する**（推奨）。例: 「動詞: 消す / 目的語: {ClassA, docs} / それ以外は触らない」。手癖の拡張解釈が起動する前に自己フェンスを置く役割
6. 実装手段に幅があるとき（発火条件 5）は auto mode でも最小案 vs 拡張案を能動確認する。「拡張で進めます、嫌なら言って」の受動的 but-out は確認とみなさない。**既定値は最小案**（字義を満たす最小実装があればそれを既定とし、拡張は明示的に選ばれたときだけ採る）

#### 先例（過去に観察された認知バイアスのパターン）

- 最小スコープ逸脱: ユーザー指示の最小範囲を逸脱して関連箇所まで巻き戻す（「ゼロ値返せ」→ required も外す）
- 意見聴取の早合点: 「意見聴取」を「修正アクション要求」と早合点する（「微妙じゃね？」→ 即修正案提示）
- 設計選択肢の汚染: 設計相談で「ユーザー既決方針に矛盾する案」「本論と直交する案」を選択肢に混ぜる
- 連鎖否定: 一軸（関数化）否定を別軸（定数化）まで連鎖否定する（「**メソッド**作る意味ある？」→ 定数化もやめてリテラル直書き）

## reviewer / refactor-advisor 指摘の事後処理

reviewer / refactor-advisor が Medium / Low 相当の指摘・提案を返してきたとき、メイン Claude（スキル本体）はそれをユーザーにそのまま転送する前に**自己照合する**。直前ターンで自分が実機検証した事実・Read した既存実装・ユーザー明示の受容判断と矛盾していないかを確認する事後ゲート。

### 発火条件

- reviewer のレポートに Medium/Low の件数が記載されているとき
- refactor-advisor から提案件数 > 0 で返ってきたとき
- ユーザーゲート（all / 番号指定 / none）を開く前
- **外部 PR Reviewer Bot（GitHub Actions の Codium PR Agent、外部 LLM レビューコメント等）の指摘を取り込む直前**。スコア・確度に関わらず、自分側で nil-safe 性・同値律・既存先例の事実関係を裏取り（必要なら explorer 委譲）してから「採用 / false positive として却下」を判定する。無批判に「PR Reviewer Bot がこう言っているので直します」と進めない

### 自己照合手順

1. **直前ターン事実との照合**: 直前ターンで Bash / Read / grep 等で「これはこうなっている」と確認した事実があれば、reviewer / refactor-advisor の指摘がその事実と矛盾していないか確認する
2. **ユーザー明示の受容判断との照合**: ユーザーが直前ターンで「これは触らない」「この仕様で OK」「これはこのままで」と明示した範囲を、reviewer / refactor-advisor が「直すべき」と言っているケースを検出する
3. **plan 段階の判断との照合**: plan 段階で USER_DECISION_REQUIRED を通って確定した方針を、reviewer / refactor-advisor が無自覚に否定する形になっていないか確認する

### 矛盾検出時の動作

- スキル本体は該当の指摘・提案を「**自己検証済み事実と矛盾**: <矛盾内容>」とラベルしてユーザーに提示する（無批判転送はしない）
- ユーザーには「reviewer の指摘 N 件のうち M 件は直前検証と矛盾」のような形で明示する
- 「直前会話の文脈を踏まえて指摘を再評価したい」というオプションをユーザーに提供する

### なぜ必要か

reviewer / refactor-advisor は単一ターンの差分だけを見ており、**直前会話で確定したユーザー方針・実機検証結果を context として持たない**。スキル本体だけがその context を持つため、ゲートを開く前に自己照合する責務はスキル本体にある。スキル本体が無批判転送すると、ユーザーは「直前ターンで話した内容を再説明する羽目になる」「これの話をしてんだよ型の叱責につながる」。本ゲートは `@~/.claude/user-feedback-protocol.md` で定義する「ユーザー指摘駆動の改善プロトコル」の **入口側** の予防策で、両者は補完関係（事後発火 ↔ 事前防止）。

### codex MCP 相談は /codex スキル経由

codex（`mcp__codex__codex`）への相談は **`/codex` スキルの手順に従う**（effort のタスク別選択・model=gpt-5.5 固定・background ラップの SSOT）。ユーザー起動だけでなく、Claude 自身がタスク途中で codex に相談すると判断したときも同様。要点だけ再掲: `mcp__codex__codex` を直接呼ぶとメイン Claude がブロックするため **必ず background Agent 経由**で呼び、応答を待つ間もメイン作業を止めない（先例 2026-06-18: フォアグラウンド待機の繰り返しをユーザーから複数回指摘）。

### コードベース探索の委譲

複数ファイルにまたがる調査・パターン探索・影響範囲の特定・既存コンポーネント列挙が必要な文脈では、自分で直接 Glob/Grep/Read/WebFetch/WebSearch を呼ばず、また Bash の `rg`/`grep`/`find`/`fd`/`ls` による広域探索も行わず、必ず explorer エージェントに委譲する（初回探索は規模に関わらず最低1体必須）。

セルフチェック（探索ツールを呼ぶ前に自問。1つでも Yes なら explorer に委譲する）:
1. 複数ファイルを跨ぐ調査・パターン探索が発生するか？
2. 「ちょっとだけ」「軽い確認だから」と自分に言い訳して直接呼ぼうとしていないか？

適用範囲・直接読んでよい例外（単一ファイルのピンポイント Read 等）・並列起動の判断軸・複数 explorer の結果統合・字義範囲ガードの詳細手順は **`~/.claude/agent-delegation.md`** を参照（探索を伴う作業に入る前に Read すること）。

### エージェント呼び出しプロトコル（PIR² 系）

PIR² 系スキル（/pir2, /pir2async, /debug, /ir 等）の内部プロトコル — チーム運用（TeamCreate）/ サブエージェント間のファイル経由受け渡し / handoff 引継ぎ / planner の能動的再探索ループ / reviewer の5観点ハイブリッド並列 / ui-ux-reviewer の追加起動（UI 変更時）/ refactor-advisor の後置運用 — の詳細は **`~/.claude/pir2-protocol.md`** を参照（**PIR² 系スキル実行時・UI 変更を含むタスク・チーム作業指示時に Read**）。各エージェント固有のガイドは `~/.claude/agents/<name>.md`。

### リファレンス実装移植タスクの忠実性ゲート（/pir2 経由でなくても適用）

タスクが**リファレンス実装（ipynb / プロトタイプ / 別言語実装 / 仕様参照実装）からの移植**（プロンプトに「準拠」「移植」「再現」「同等」「忠実に」、または `REFERENCE_PATH` 明示）と判明したら、`/pir2` 等のワークフローを経由せず**直接実装する場合でも**、メイン Claude が次を必ず適用する（移植ゲートの詳細ルールは `reviewer.md` / `explorer.md` に既に SSOT 化済み。直接タスクだとそれらが発火しない盲点を埋める橋渡し）:

1. **explorer に完全抽出を指示**: `~/.claude/agents/explorer.md`「リファレンス実装からの移植タスクでの一括抽出網羅性」（プロンプト原文の**構造＝Step/優先順位/セクション見出し**・スキーマ・**料金/modality 別単価**・**全 rendering 分岐の空フォールバック文言**・description まで 1 回で全件抽出。`flat-text 計上` 等の単純化を移植先で勝手にしない）を起動プロンプトに明記する。
2. **reference-fidelity reviewer を必須起動**: `~/.claude/agents/reviewer.md`「リファレンス移植タスクでの reference-fidelity 必須化」に従い `REVIEWER_ROLE=reference-fidelity` ＋ `REFERENCE_PATH=<絶対パス>` を渡して逐語照合させる。手動で reviewer を並列起動する場合も必ず観点に含める。
3. **既存リポ慣習で reference 差異を却下しない**: reviewer の reference 差異指摘を「既存先例 X も同じだから」で弾かない（既存先例自体が乖離している場合がある）。

> ℹ️ あるタスク類型の観点・ルールを追加する前に、まず該当エージェント定義（`reviewer.md` / `explorer.md`）と本 CLAUDE.md が既に当該類型をカバーしていないか確認する。既存 SSOT にあるのに memory / プロジェクト docs へ重複追記しない（二重管理は SSOT 違反）。
