# Claude Code 設定

## 出力言語
- すべての出力・回答は**日本語**で行う

## グローバルファイルの汎用性ルール

グローバルスコープのファイル（`~/.claude/CLAUDE.md`、`~/.claude/agents/`、`~/.claude/skills/`）は **全プロジェクト共通** で使われる。特定プロジェクトに依存する内容を混入させてはならない。

- **禁止**: プロジェクト固有の名前（クラス名、テーブル名、API エンドポイント名、`FooUsecase` のような具体名）
- **許可**: 汎用的な概念（「usecase」「repository」「controller」等のパターン名）や例示としての仮名（`XxxService` 等）
- プロジェクト固有の情報はそのプロジェクトの `.claude/CLAUDE.md` またはプロジェクトメモリ（`~/.claude/projects/` 配下）に書くこと
- retrospector がエージェント定義を更新する際もこのルールを厳守すること

## Git ルール

- `git add -A` や `git add .` は **絶対に使わない**。コミット対象のファイルを個別に指定すること
- マージコミットではコンフリクトしたファイルだけを `git add` する

## tester エージェントのルール

- テストデータが不足している場合は自分でテストデータを投入すること。「DBにデータがないので確認不可」は許容しない
- テストデータを投入して正常系フロー全体を検証し、テスト後はデータをクリーンアップすること

## retrospector エージェントのルール

- 役割は振り返りと改善提案のみ。ファイルのリネーム、コード修正、リファクタリングなどのコード変更は一切禁止
- 改善が必要と判断した場合はレポートに記載して返すだけにすること（エージェント定義・スキル定義の更新は許可される）

## 書式ルール

- `**太字**` などの強調記法を多用しない。すべての内容が等しく重要なので、特定の箇所だけを強調しない

## tech-validator エージェントの使用ルール

### 使うタイミング
以下の状況では **必ず** `tech-validator` エージェントを起動してからライブラリを決定すること：

- 新しいライブラリ・フレームワークをプロジェクトに追加するとき
- 既存依存関係の更新・置き換えを検討するとき
- 同種のライブラリが複数存在し、どれを選ぶか判断が必要なとき
- 「○○を実現するのに適したライブラリは？」と聞かれたとき

### 禁止事項
- バージョンを記憶や推測で記載しない（必ず `npm show` で確認）
- tech-validator を経ずに独断でライブラリを選定しない
- 選定レポートなしに `npm install` を提案しない

## エージェントチーム運用ルール
- 「エージェントチーム」「チームで作業」等の指示があった場合、Subagentsではなく Agent Teams 機能を使うこと
- エージェント間で共有コンテキストを持ち、相互にメッセージングできる構成にすること
- 単に複数のサブエージェントを順次起動するだけの構成は「チーム」とみなさない

## 問題の迂回禁止
- 正しい設計に従ったコードが期待通りに動かない場合、勝手に別の手段で迂回しない
- ツールやインフラが使えない場合も、勝手に代替手段で進めない
- いずれの場合も原因を報告してユーザーの判断を仰ぐ

## スコープ厳守
- 機能実装時は指示されたスコープを厳守する。明示的に依頼されていない隣接モジュールへの変更を勝手に行わない

## コードベース探索の委譲ルール

Opus が既存コードの調査（ファイル構造の把握、パターン調査、影響範囲の特定など）を行う必要がある場合、自分で直接 Glob/Grep/Read を呼ばず、explorer エージェント（Sonnet）に委譲すること。

- 調査対象が独立した複数の領域にまたがる場合は、最大3体まで並列起動する
- 各エージェントには調査対象と期待する出力を具体的に指示する
- Opus は返ってきた探索レポートをもとに判断・回答・実装を行う
- 単一ファイルのピンポイント確認（パスが既知で内容を1箇所だけ見る）は直接 Read して良い

## デバッグの進め方
- デバッグ時は、まずユーザーが報告した実際のフロー・パスを確認する。コード分析に入る前に、どのフローが問題になっているか明確にし、不明な場合は質問して確認する
