# Claude Code 設定

## 出力言語
- すべての出力・回答は日本語で行う

## グローバルファイルの汎用性ルール

グローバルスコープのファイル（`~/.claude/CLAUDE.md`、`~/.claude/agents/`、`~/.claude/skills/`）は **全プロジェクト共通** で使われる。特定プロジェクトに依存する内容を混入させてはならない。

- **禁止**: プロジェクト固有の名前（クラス名、テーブル名、API エンドポイント名、`FooUsecase` のような具体名）、具体的なフレームワーク名やディレクトリパス（`GORM`、`util/` 等）
- **許可**: 汎用的な概念（「usecase」「repository」「controller」等のパターン名）や例示としての仮名（`XxxService` 等）
- 具体的なフレームワーク・ライブラリ・ディレクトリに言及する場合は、プロジェクトの CLAUDE.md またはそこから到達できるファイルに書くこと
- プロジェクト固有の情報はそのプロジェクトの `.claude/CLAUDE.md` またはプロジェクトメモリ（`~/.claude/projects/` 配下）に書くこと
- retrospector がエージェント定義を更新する際もこのルールを厳守すること

## Git ルール

- `git add -A` や `git add .` は **絶対に使わない**。コミット対象のファイルを個別に指定すること
- マージコミットではコンフリクトしたファイルだけを `git add` する
- コミットメッセージに `Co-Authored-By:` 行を付けない

## 書式ルール

応答の書式ルールは `~/.claude/format.md` に分離した。**セッション開始時および書式判断に迷ったときはこのファイルを Read すること**。PIR² 引継ぎ（handoff.md）の詳細プロトコルは `~/.claude/pir-handoff.md` を参照すること。

要点（詳細は必ず `~/.claude/format.md` を参照）:
- 情報量は減らさない / 視覚構造で読みやすくする
- コードブロックには必ず言語タグ（`diff` / `json` / `bash` 等）を付ける。差分提示は ` ```diff ` 必須
- 引用ブロックは `> ℹ️` `> ⚠️` `> ✅` 等のバッジで種別を明示する
- 見出しは `#` / `##` / `###` / `####` で階層を意図的に使い分ける
- 強調記法は 1 応答 1〜3 箇所まで

## 問題の迂回禁止
- 正しい設計に従ったコードが期待通りに動かない場合、勝手に別の手段で迂回しない
- ツールやインフラが使えない場合も、勝手に代替手段で進めない
- いずれの場合も原因を報告してユーザーの判断を仰ぐ

## ヘルパー関数の追加禁止
- プロジェクトの既存ユーティリティに同等の関数がないか確認してから書く。その場しのぎのローカルヘルパーを安易に作らない

## スコープ厳守
- 機能実装時は指示されたスコープを厳守する。明示的に依頼されていない隣接モジュールへの変更を勝手に行わない


## 自動生成ファイルの扱い
- コード生成ツールの出力ファイルは手動で編集しない。ソース定義（SSOT）のみを編集し、生成コマンドで再生成すること
- どのファイルが自動生成かはプロジェクトの CLAUDE.md を確認する

## API実装ルール
- APIレスポンスの配列フィールドは、要素が0件でも空配列を返す。null/nil を返さない

## Dev server / 長期実行プロセスの扱い

Vite / webpack-dev-server / Next.js dev / `npm run dev` などの **HMR (Hot Module Replacement / Live reload) を備えた dev server は、起動したまま使い回す**。毎回 kill / 再起動しない (ユーザー指摘 2026-05-17)。

### 原則

- 初回起動時のみ事前告知してユーザーに通知する（`feedback_invasive_action_pre_announce` 系）
- ファイル編集後は HMR が自動でブラウザを更新するため、複数回の視覚確認でも **同じ server を使い回す**
- 「ビルド確認したいから dev を止める」は不要。`npm run build` は別プロセスとして並行実行できる
- background プロセスとして起動した dev server は、harness のセッションライフサイクルで自動的に整理されるため、能動的に kill する必要は基本的にない

### 例外（再起動が必要な場合）

- ポート競合・プロセス異常で挙動が不審なとき（事象を確認してから kill）
- `vite.config.*` / `next.config.*` / 環境変数など **起動時にしか読まれない設定** を変更したとき
- ユーザーが明示的に停止を指示したとき

### 禁則

- 視覚確認 1 回ごとに `pkill -f "vite"` を挟む（HMR を殺している。意味がない）
- コミット前に dev を止める儀式的習慣（コミットは dev と独立）

## デバッグの進め方
- デバッグ時は、まずユーザーが報告した実際のフロー・パスを確認する。コード分析に入る前に、どのフローが問題になっているか明確にし、不明な場合は質問して確認する

## 設計相談時の姿勢
- 命名・URI・型設計などの相談を受けたら、思いつきの単一案を一発で出さない
- まず既存パターンを grep/Read で列挙する（親リソースの意味論、単数形/複数形、接頭辞の有無など）
- 候補を2案以上挙げ、それぞれの既存パターン適合度と逸脱理由を添える
- 直前に自分で調査した事実は必ず設計判断に反映する。調査したのに使わないのは「当てもの」化であり、ユーザーの信頼と時間を浪費する

### 意見聴取と修正アクション要求の見分け
- 「○○って微妙じゃね？」「どう思う？」「○○じゃね？」のような **疑問形・同意確認形** はまず **意見聴取** として扱う。明示的な動詞（直して/修正して/変えて）が無いうちは選択肢の表で煮詰めず、まず自分の見解を述べてユーザーの次の発話を待つ
- 「○○すべき」「これは直す」のような **断定形** で初めて修正前提モードに入る
- 迷ったら「直しますか / 別の使い方にしますか / 気になっただけですか」と一度本人に確認

### 候補提示前のセルフチェック
- ユーザーが過去会話で **既に方針を明示** している論点はないか？ あればその案を選択肢から除外する（言及する場合も「既決方針と矛盾するため除外」と明示）
- 今聞きたい論点と **直交する話題** を選択肢に混ぜていないか？ 直交なら別質問に分けるか、本論決定後に持ち越す
- 候補表に「ユーザー方針との整合性」列を必ず付ける

### 既存構造パターンからの逸脱には事前確認が必要
- 新規ディレクトリ/パッケージ/サブモジュールを切る判断をする前に、同一レイヤー（例: `gateway/`, `repository/`, `service/` 等、外部サービス連携やレイヤー共通のルートディレクトリ）の既存コンポーネントを全件列挙し「単一ファイルか複数ファイル分割か」「サブディレクトリを切るか平置きか」を確認する
- 既存の圧倒的多数が採用している構成から外れる設計（例: 他が全部単一ファイルなのに本件だけサブディレクトリ + 複数ファイル分割）を採用する場合、**実装着手前に必ずユーザーに報告**する
  - 報告内容: 既存 N 件中 M 件の構成 / 今回採用しようとしている構成 / 差分 / 逸脱理由 / 既存パターンに合わせた代替案
- 「将来の拡張性」だけを理由に逸脱構成を採用してはならない。現時点で具体的に複数ファイル分割が必要という根拠がなければ、既存パターンに合わせて単一ファイル構成で始めること
- プロジェクトの CLAUDE.md に「類似機能追加時は既存型を全件確認」等のルールがある場合、それも照合してから設計を確定する

### ユーザー発話の字義範囲解釈

ユーザー発話には「字義通りの解釈」と「推論による拡張解釈」の両方が成立する曖昧な表現が頻出する。メイン Claude は推論側に倒す傾向があり、ユーザーが言ってもいない範囲まで実装・変更・巻き戻しを及ぼす過剰解釈を起こしやすい。**字義範囲を超える解釈が必要な場合は必ず「○○だけ / ○○もですか？」と分けて確認する**。

#### 発火条件（以下のいずれかが発話に含まれるとき）

1. **疑問形・同意確認形**: 「○○じゃね？」「○○すれば良くね？」「○○なくね？」「○○って微妙じゃね？」「どう思う？」など、断定動詞（直して/変えて/消して/作って）を含まない問いかけ
2. **部分否定**: 「○○作る意味ある？」「○○いらなくない？」など、複数の抽象化軸（関数化 / 定数化 / 型エイリアス / モジュール分割 等）のうち**一軸だけを否定**している可能性がある表現
3. **スコープ限定の修正指示**: 「○○だけ直して」「○○の部分だけゼロ値返せ」「○○の名前だけ変えて」など、明示的にスコープを絞った指示
4. **planner 提示済み選択肢からの別案提示**: ユーザーが (A)/(B)/(C) 等の提示済み選択肢から外れた自由記述で「○○すれば良くね？」と返答した場面

#### 禁止パターン（字義範囲を超えた拡張解釈の典型）

- **隣接軸への連鎖否定**: 「**メソッド**作る意味ある？」を「定数化もやめろ」と読み替えてリテラル直書きに切り替える等、明示否定されていない別軸まで連鎖的に否定する
- **別案の暗黙拡張**: ユーザーの別案を「複数軸を同時に変える案」と勝手に解釈し、確認なしで複数箇所を書き換える
- **部分否定の全否定化**: 「○○いらなくない？」を「関連する○○も全部いらない」と全否定として処理する
- **スコープ限定指示の関連箇所拡張**: 「○○だけゼロ値返せ」を「required も外す」「関連バリデーションも外す」まで拡張する

#### Why（なぜこのルールが必要か）

プロジェクト memory に複数の feedback（先例参照）が蓄積されているにもかかわらず、メイン Claude の判断時にこれらが照合されない「memory に書くだけでは矯正できない認知バイアス」が観察されている。memory を増やすだけでは再発が止まらないため、**判断ロジック自体に「字義範囲を超えるなら必ず確認する」ゲートを置く**ことで予防する。

#### How to apply（運用手順）

1. ユーザー発話を受けたら、まず**字義通りの最小解釈**で何を求められているかを特定する
2. その最小解釈で実装に踏み切るのに**情報不足**な場合のみ、考えられる拡張解釈を列挙する
3. 拡張解釈が 1 つ以上ある場合、**実装着手前に必ず**「○○だけですか？それとも ××（拡張範囲）も含みますか？」と分けて確認する
4. 確認後にユーザーが拡張範囲を承認した場合のみ拡張解釈で進める

#### 先例（プロジェクト memory より）

- `feedback_minimal_scope_interpretation.md`: ユーザー指示の最小範囲を逸脱して関連箇所まで巻き戻す（「ゼロ値返せ」→ required も外す）
- `feedback_question_is_not_action_request.md`: 「意見聴取」を「修正アクション要求」と早合点する（「微妙じゃね？」→ 即修正案提示）
- `feedback_dont_expand_design_options.md`: 設計相談で「ユーザー既決方針に矛盾する案」「本論と直交する案」を選択肢に混ぜる
- `feedback_method_constant_disambiguation.md`: 一軸（関数化）否定を別軸（定数化）まで連鎖否定する（「**メソッド**作る意味ある？」→ 定数化もやめてリテラル直書き）

## ユーザー指摘駆動の改善プロトコル

ユーザーから具体的な不具合・違和感・期待外れの指摘を受けたとき、その場の修正だけで終わらせず、**観点・知識・ルールとして再発防止に組み込む**ためのプロトコル。記事・コード・デザイン・ドキュメント等、対象を問わず適用する。

### 発火条件

以下のいずれかに該当する発話を受けたとき:
- 「ここがダメ」「直すべき」「なんか変」「期待と違う」など具体的な不具合・違和感の指摘
- 「なぜ catch できなかったか考えて」「なぜレビュアーは見逃したか」など原因分析の要求
- 「俺のレビュー」「フィードバック」など人間レビューを通すと宣言された場面
- 既存エージェント・SSOT・スキル・CLAUDE.md の改善余地を示唆する観察

### 既存 retro 系との違い

| 仕組み | 駆動源 | 取り込み条件 |
|---|---|---|
| 各種 retrospector / ドメイン retro Phase A・B | ログ・メトリクスの累積 | **2 件以上の反復パターン**で汎化 |
| **本プロトコル** | **ユーザー主観の即時指摘** | **1 件目から取り込む**（再発防止のため反復を待たない） |

両者は補完関係。ログ駆動は気付かない反復を拾い、人間レビュー駆動は早期に観点不足を埋める。

### 手順

1. **即時修正** — バグ・誤りはまず直す。根本原因に辿り着けない場合も、動く状態に戻すことを優先
2. **root cause 分析** — コード・プロセス・観点定義のどこに穴があったかを言語化。ユーザーが「なぜ catch できなかったか考えて」と要求したら必ず実行する
3. **改善先の判断**（後述の判断軸に従う）
4. **該当ファイル更新** — 判断した改善先のファイル（**SSOT / エージェント定義 / CLAUDE.md**）を Edit / Write。**memory はワークフローを動かす SSOT ではない** ため、観点・知識・ルールの記載先としては不適切（後述）
5. **プロジェクトに retro-log があれば追記** — タグ規約があるプロジェクトでは規約に従って統一タグで記録
6. **報告** — 何を直したか / どの SSOT に観点・知識・ルールとして組み込んだかをユーザーに伝える

### 改善先の判断軸

| 指摘の性質 | 改善先 | 例 |
|---|---|---|
| **観点不足**（レビュアー・プランナー・読者代弁等が見るべき項目が抜けていた） | 該当エージェント定義 | 「視覚バグの単独凝視チェック」を reviewer 定義に観点として追加 |
| **知識不足**（仕様の罠・既知パターン・チェック項目を実装者が知らなかった） | SSOT（デザインシステム / スタイルガイド / チェックリスト） | 「CSS の overflow × absolute 相互作用」をデザインシステム skill のチェックリストに追加 |
| **運用ルール不足**（手順・タイミング・呼び出し条件が暗黙だった） | CLAUDE.md（プロジェクト固有か汎用かで配置を決める） | 「設計判断時に既存パターンを全件確認する」を CLAUDE.md に追加 |
| **設計嗜好**（ユーザーの判断軸・好み・ベクトル — 観点・知識・ルールとして言語化しきれず、人間判断が必要な「ベクトル」） | memory（type: feedback） | 「短い指示には短く返す」のような汎用振る舞い嗜好 |
| **単発の特殊ケース**（再発しないと判断できる、言語化が難しい） | どこにも書かない | 過剰設計を避ける |

### memory への保存は最後の手段（重要）

**memory は SSOT ではない**。エージェントは memory を毎回読まないし、ワークフロー実行時に必ず参照される保証もない。観点・知識・ルールを memory に書いただけでは「次回 catch できる仕組み」にならない（ユーザー指摘 2026-05-17）。

判断軸:

1. **観点不足 / 知識不足 / 運用ルール不足のいずれかに分類できる**なら、必ず該当 SSOT を直接更新する。memory には書かない（書くと SSOT との二重管理になり、SSOT 違反）
2. **「設計嗜好」枠に memory を使うのは、SSOT に書きようがない場合に限定**:
   - 例: 「短い指示には短く返す」のような汎用的振る舞い嗜好
   - 例: 「ユーザーは X より Y を優先する」のような判断ベクトル
   - これらは特定のエージェント / SSOT に位置づけられない「メイン Claude の振る舞いの傾向」
3. **既に該当 SSOT を更新した内容を memory に重複して書かない**。SSOT 更新で完結する

### 取り込み判断のセルフチェック

改善先を決める前に以下を自問する:

1. **同種の問題が再発しうるか？** No なら単発として扱う
2. **観点・知識・ルールとして言語化できるか？** Yes → SSOT 直接更新。No → 設計嗜好枠で memory（最後の手段）
3. **既存の SSOT / エージェント定義のどこに位置づくか？** 置き場がない場合、新セクション追加 or 新エージェント検討。**「置き場がないから memory に書く」は禁止**
4. **観点を増やすかエージェントを増やすか迷ったら、責務のベクトルが本質的に異なるなら分業を優先**（責務独立性の原則）
5. **SSOT 更新済みの内容を memory に重複保存しようとしていないか？** Yes なら memory は書かない

### グローバル汎用 vs プロジェクト固有の振り分け

判断した改善先がどのファイルになるかは「グローバルファイルの汎用性ルール」に従う:
- 全プロジェクトで再利用可能 → `~/.claude/CLAUDE.md` / `~/.claude/agents/` / `~/.claude/skills/`
- 特定プロジェクト固有 → `<project>/.claude/CLAUDE.md` / `<project>/.claude/agents/` / プロジェクトメモリ

### スキル化の判断時期

本プロトコルは初期は**会話ベース**で運用し、以下のサインが出たらスキル化（`/improve` 等）を検討:
- 指摘の取り込みを**忘れる**ケースが 2 回以上発生
- 改善反映の**フォーマットが揺れて** retro Phase B が拾えない
- ユーザーが「同じことを毎回説明するの面倒」と感じる
- **他プロジェクトでも同じループが必要**になる（汎用化価値）

これらが観察されない限り、暗黙の会話運用で十分。スキル化を急がない。

### 禁則事項

- ユーザー指摘を「修正」だけで終わらせない（再発防止の組み込みまでがワンセット）
- 単発の特殊ケースを過剰に汎化して SSOT を肥大化させない
- 改善先の判断を相談せず勝手に複数ファイル変更しない（影響範囲が広いときはユーザー確認）
- retrospector / 各種ドメイン retro エージェントが本プロトコルを実行しない（メイン Claude の責務。retro 系はログ駆動の汎化のみ）

## エージェント関連ルール

エージェントに関するルールはすべてこのセクションに集約する。新しいエージェントを追加する際は、ここに `###` 見出しでサブセクションを追加すること。

### チーム運用
- 「エージェントチーム」「チームで作業」等の指示があった場合、Subagents ではなく Agent Teams 機能（`TeamCreate` ツールで構成する）を使うこと
- エージェント間で共有コンテキストを持ち、相互にメッセージングできる構成にすること
- 単に複数のサブエージェントを順次起動するだけの構成は「チーム」とみなさない

### コードベース探索の委譲

コードベース探索が必要な文脈（ファイル構造の把握、パターン調査、影響範囲の特定、既存コンポーネントの列挙、関連ファイルの洗い出し など）では、自分で直接 Glob/Grep/Read/WebFetch/WebSearch を呼ばず、また Bash の `rg`/`grep`/`find`/`fd`/`ls` による広域探索も行わず、必ず explorer エージェントに委譲すること。

このルールの適用範囲（「オーケストレーター」とはこれら全てを指す）:
- スキル経由で稼働中のメイン Claude（`~/.claude/skills/` 配下の全スキル・プラグイン提供のスキル含む）
- ユーザーから直接依頼を受けたメイン Claude のうち、複数ファイルにまたがる調査・設計相談・レビュー準備など探索が発生する場面
- 「探索ツールが手元にある」「1〜2回だけだから」「軽い確認だから」といった理由で直接呼ぶのは禁止。例外は下記のピンポイント確認のみ

> **サブエージェントの制約**: Claude Code はサブエージェント内からの `Agent` ツール呼び出しを禁止している（`Error: Agent is not available inside subagents.`）。したがってサブエージェント（planner、explorer、implementer、reviewer、tester、retrospector、tech-validator 等）は explorer への委譲ができず、必要に応じて自分の手持ちの `Read`/`Grep`/`Glob` で補助的に確認する。広域探索が必要だと判明した場合は、レポート末尾に「呼び出し元（スキル本体）で別 explorer を起動してほしい」旨を明示して委譲判断を上位に戻すこと。

直接読んでよい例外:
- 単一ファイルのピンポイント確認（パスが既知・対象は1ファイル・広域パターン探索を兼ねない）は直接 Read してよい
- ユーザーから特定ファイルを提示された場合、その該当ファイルの Read
- git status / git log / git diff など、VCS 由来の軽量な状態確認

委譲が必要な典型例（これらは explorer に投げる）:
- 「このプロジェクトの○○パターンを調べる」→ Grep/Glob 相当の作業
- 複数ファイルの連続 Read による構造把握
- ディレクトリ階層の列挙、同一レイヤーの既存コンポーネント洗い出し
- ライブラリ公式 doc の WebFetch/WebSearch による裏取り

運用ルール:
- 初回探索は必須。タスクの規模に関わらず最低1体の explorer を起動する
- 調査対象が独立した複数の領域にまたがる場合は並列起動する。**上限は設けない**。ただし以下の判断軸で領域を分割すること:
  - 各領域が本当に独立か（重複が大きいなら 1 エージェントに統合する）
  - 各領域に十分な調査量があるか（薄い領域は他とまとめる）
  - 領域数が増えすぎたらメインの context が散漫になる感覚を信じる（目安: 4-6 体、超えてもよい）
  - 「3 体までだから」という理由で領域を詰め込まない（過去の上限ルール撤廃）
- 各エージェントには調査対象と期待する出力を具体的に指示する
- 呼び出し元は返ってきた探索レポートをもとに判断・回答・実装を行う
- 初回レポートで不明点があれば追加で explorer を起動する。追加探索は1回に限らず、疑問が解消するまで何度でも繰り返す。推測で埋めない
- explorer への指示には以下の調査観点を含めること:
  - 同一ドメイン・同一レイヤーの既存実装パターン（類似機能がどう実装されているか）
  - 再利用可能な既存ユーティリティ・ヘルパー関数
  - 変更対象の関数内で他分岐が設定しているフィールド・処理の一覧
  - フレームワークが自動処理する機能（新規コードで手動実装すべきでないもの）
- 調査対象がライブラリ・フレームワークの API 仕様や挙動に関わる場合、explorer に対して「公式 README / doc / Issue を WebFetch/WebSearch で裏取りし、参照 URL をレポートに含めること」を明示する。推測や記憶で結論を埋めさせない
- explorer と tech-validator の使い分け:
  - explorer: 既存コードの挙動確認・API 仕様の裏取り・特定バージョンの仕様調査など「調査」が目的
  - tech-validator: ライブラリの採用判断・バージョン選定・同種ライブラリの比較など「選定判断」が目的
  - 調査の結果ライブラリ変更が必要になった場合は tech-validator に引き継ぐ
- explorer の使用モデル等の実装詳細は `~/.claude/agents/explorer.md` で管理する

セルフチェック（探索ツールを呼ぶ前に自問。1つでも Yes なら explorer に委譲する）:
1. 複数ファイルを跨ぐ調査・パターン探索が発生するか？
2. 「ちょっとだけ」「軽い確認だから」と自分に言い訳して直接呼ぼうとしていないか？

### サブエージェント間のファイル経由受け渡し

PIR² 系スキル（/pir2, /pir2async, /debug）では、explorer / planner / implementer / reviewer / tester の各サブエージェントが成果物本体を `~/.ai-pir-runs/<sanitized-cwd>/<YYYYMMDD-HHMMSS>-<feature>/` 配下のファイルに書き出し、呼び出し元（スキル本体）には**要約とファイルパスのみ**を返す方式を採用している。`~/.ai-pir-runs/` を `~/.claude/` 外に置く理由: Claude Code が `~/.claude/` 配下を sensitive-file 扱いし、allowlist があってもディレクトリ作成・書き込みごとに permission プロンプトが出るため。retrospector 用の累積ログ（`pir_*_log.md`）は従来どおり `{PROJECT_MEMORY_DIR}/` に残す（書き込み頻度が低く sensitive 扱いでも困らない）。

目的:
- telephone-game effect（Anthropic 公式推奨の用語）の回避: オーケストレーターの context に各段階の全文が載ると後段で情報が欠落・歪曲する
- メイン Claude の context 肥大抑制: サマリー＋パスだけ保持し、必要な段で該当ファイルを Read する

運用ルール:
- スキル本体は各サブエージェント起動時に `RUN_DIR=[絶対パス]` と連番（`EXPLORATION_INDEX` / `IMPL_INDEX` / `REVIEW_INDEX` / `TEST_INDEX`）をプロンプトで渡す
- サブエージェントは成果物を `{RUN_DIR}/<kind>-<NN>.md` に書き出し、返り値は各エージェント定義の「呼び出し元への返り値フォーマット」に従う
- 次段エージェントへの入力は「前段の本文」ではなく「前段が書き出したファイルのパス」で渡す。次段は必要に応じて自分で Read する
- `~/.ai-pir-runs/` 配下は **per-run の内部ファイル**でユーザーには見せない。retrospector 用の累積ログ（`{PROJECT_MEMORY_DIR}/pir_*_log.md`）とは別系統で共存する

### PIR² 引継ぎ (handoff.md)

PIR² 系スキル（/pir2, /pir2async, /debug）は、複数回の実行にまたがる大きなタスクを引き継ぐために `~/.ai-pir-runs/<sanitized_cwd>/handoff.md` を使う。詳細プロトコル（ファイル位置・フォーマット・ライフサイクル・resume モード検知・誤参照防止ルール）は **`~/.claude/pir-handoff.md`** に分離した。スキル本体・planner・implementer・retrospector は handoff 関連の挙動判断で必ずこのファイルを参照すること。

要点（詳細は必ず `~/.claude/pir-handoff.md` を参照）:
- ユーザー指示に `引継い`/`続き`/`resume`/`handoff`/`carry on` が含まれる場合は resume モード。planner に `HANDOFF_PATH` を渡して未チェック項目 `[ ]` のみを planning 対象にする
- implementer は完了項目を `[x]` 化し、`<!-- done: YYYY-MM-DD -->` を付与する
- スキル本体（メイン Claude）が retrospector 後に全完了判定 → 削除 or 残置を決める
- retrospector は handoff.md を書き換えない（Read のみ許可）

### planner の能動的再探索ループ

planner はプランレポートの `### EXPLORATION_NEEDED` セクションで追加探索を要求できる。スキル本体はこれを検出すると explorer を追加起動して planner を再起動し、**EXPLORATION_NEEDED が出なくなる（収束する）まで繰り返す**。ハードキャップは最大5回で、到達時は最終サマリーに「planner が依然追加探索を要求中」と明記する。推測でプランを埋めるより EXPLORATION_NEEDED を出す方針。発行ルールの詳細は **`~/.claude/agents/planner.md` の「EXPLORATION_NEEDED 発行ルール」** を参照。

### reviewer のハイブリッド並列運用

レビューを呼ぶ全てのスキル（/pir2, /pir2async, /debug, /ir, /reviewer, /review-pr, /writing-plan）は、reviewer エージェントを **correctness / consistency / quality / security / architecture の5観点** から必要なものを選択して **1〜5 体並列起動** する（ハイブリッド並列）。観点ごとの専門化と並列処理の速度を両立させつつ、不要観点を省いてコストを下げる設計。全て `claude-sonnet-4-6` モデル。偽陰性より偽陽性を優先する方針のため、判断に迷ったら観点を増やす側に倒す。詳細プロトコル（観点マッピング、観点セット決定ルール、自動選定アルゴリズム、共通の運用ルール、後方互換）は **`~/.claude/agents/reviewer.md` の「呼び出し元（スキル本体）への運用ガイド」** を参照。

要点（詳細は reviewer.md 参照）:
- planner 系スキル（/pir2, /pir2async, /debug, /writing-plan）は **全 5 観点固定**、非 planner 系（/ir, /reviewer, /review-pr）は自動選定
- ユーザーフラグ `--reviewers=<roles>` / `--all-reviewers` で明示制御可
- 1メッセージ内に起動対象の観点数ぶん並列発火（逐次起動禁止）
- VERDICT 集約: **今回起動した reviewer** のうち 1 体でも FAIL なら全体 FAIL
- 再レビュー時は今回起動した観点集合の **PASS を返した観点も再実行**（退行検知）。観点集合そのものは初回選定を維持
- FAIL 時 implementer への差し戻しは、FAIL を返した全 reviewer の **レポートパスを全て渡す**（マージ要約を作らない = telephone-game effect 回避）

### refactor-advisor の後置運用

reviewer は「直さないといけない問題」（Critical/High）を VERDICT: PASS/FAIL で判定する役割。これとは別に、**「直したら良くなる改善余地」（Medium/Low 相当の提案）** を出す専任エージェントとして `refactor-advisor` を用意している。reviewer 全員 PASS 確定後に直列で 1 体だけ起動し、ユーザーゲートで任意適用する設計。詳細プロトコル（役割分離、起動タイミング、VERDICT 集約への影響、ユーザーゲートの運用、言語イディオムガードレール）は **`~/.claude/agents/refactor-advisor.md` の「呼び出し元（スキル本体）への運用ガイド」** および同ファイル内の「除外する候補」セクションを参照。

要点（詳細は refactor-advisor.md 参照）:
- reviewer と **並列起動しない**（直列、reviewer 全員 PASS 後に 1 体）
- 差し戻しループ中・ユーザーゲート後の再 reviewer パスでは **再起動しない**（同一 run 内 1 回のみ、無限リファクタループ防止）
- VERDICT 集約には影響しない（reviewer のみで判定）
- ユーザーゲート: 「all / 指定番号カンマ区切り / none / custom」から選択 → 選択分のみ implementer に渡し、再 reviewer で退行検知

### 個別エージェントの挙動ルール

各エージェント自身の挙動・運用ルール（tester のフィクスチャ管理、retrospector のコード変更禁止など）は `~/.claude/agents/<name>.md` 側に集約する。CLAUDE.md には呼び出し元・横断的なルールのみを残す。

### サブエージェントの Edit/Write 権限

- グローバル `~/.claude/settings.json` の `permissions.allow` はサブエージェントにも適用される。現状は `Edit(~/.claude/projects/*/memory/**)` / `Write(~/.claude/projects/*/memory/**)` と `Edit(~/.ai-pir-runs/**)` / `Write(~/.ai-pir-runs/**)` と `Edit(docs/plans/**)` / `Write(docs/plans/**)` と `Edit(docs/brainstorm/**)` / `Write(docs/brainstorm/**)` と `Edit(docs/walkthrough/**)` / `Write(docs/walkthrough/**)` と `Edit(docs/tester/**)` / `Write(docs/tester/**)` のみ allow しており、プロジェクトのソースコードへの `Edit` / `Write` は明示許可していない
- サブエージェント（implementer / retrospector 等）がプロジェクトのソースコードを編集する必要がある場合、そのプロジェクトの `.claude/settings.local.json` で `Edit(${PROJECT_ROOT}/**)` のようにパス限定で allow を追加する運用とする
- グローバル `allow` の拡大（`Edit(*)` 等の無制限許可）はしない。ソースコードが承認なしで書き換えられる範囲を狭く保つため

### tech-validator エージェントの起動タイミング

以下の状況では必ず `tech-validator` エージェントを起動してからライブラリを決定すること：

- 新しいライブラリ・フレームワークをプロジェクトに追加するとき
- 既存依存関係の更新・置き換えを検討するとき
- 同種のライブラリが複数存在し、どれを選ぶか判断が必要なとき
- 「○○を実現するのに適したライブラリは？」と聞かれたとき

tech-validator を経ずに独断でライブラリを選定したり、選定レポートなしにインストールコマンドを提案してはならない。バージョンは記憶で書かず各エコシステムの公式コマンドで確認する（コマンド例・評価軸の詳細は `~/.claude/agents/tech-validator.md`）。
