# Ideal State

デザインシステムの「あるべき姿」を定義する。
スタック非依存の原則と、スタック別の検証基準を分けて記述する。

audit時はこのファイルを正として現状を評価せよ。

---

## 原則（スタック非依存）

### 1. SSOTが存在し機能している

- [ ] プロジェクトルートに `design-system.config.*` が存在する
- [ ] SSOTにデザイントークン（色・スペーシング・タイポグラフィ）が定義されている
- [ ] スタイリングファイルがSSOTを参照している（直接importまたはビルドツール経由）
- [ ] SSOTにアンチパターンが明記されている
- [ ] トークンが少なくとも2層（値の定義とセマンティックな役割割り当て）で構造化されている
  - 例: `blue-500 = #3b82f6`（値） → `primary = blue-500`（役割）
  - コンポーネント固有のトークン層（3層目）は、UIライブラリの公開やマルチテーマ対応が必要な場合に検討する

### 2. トークンの値が適切に設計されている

- [ ] カラートークンの値がWCAG 2.1 AAのコントラスト比（4.5:1 通常テキスト、3:1 大テキスト）を満たしている
- [ ] タイポグラフィのサイズ値がrem等の相対単位で定義されている（pxのハードコードではない）

### 3. トークン外の値が使われていない

- [ ] ハードコードされた色値（`#3b4a6b`, `rgb(...)` 等）がない
- [ ] ハードコードされたpx値（スペーシング・フォントサイズ）がない
- [ ] SSOTで定義されていないトークン名が使われていない

### 4. コンポーネントが規則に従っている

- [ ] 命名規則が統一されている
- [ ] コンポーネントが定められたディレクトリに配置されている
- [ ] インラインスタイルが使われていない（例外はSSOTに明記されている）
- [ ] 同じ見た目・ふるまいを持つコンポーネントが複数存在しない（重複コンポーネント）

### 5. ふるまいとスタイルが分離されている

- [ ] インタラクション（開閉・フォーカス・キーボード操作）がスタイルと混在していない
- [ ] アクセシビリティ属性（aria-*）が適切に付与されている

### 6. レスポンシブ・テーマ対応が一貫している

- [ ] レスポンシブ対応がある場合、ブレークポイントがSSOTで定義されている
- [ ] コンポーネントごとにブレークポイントの値がバラバラになっていない
- [ ] ダークモード/テーマ切り替えがある場合、色トークンがセマンティック（`primary`, `background`等）に定義され、テーマごとの値解決がSSOTで管理されている
- [ ] テーマ対応がない場合でも、将来のテーマ追加を阻害するハードコードがない

### 7. デザインシステムが成長できる状態にある

- [ ] 新しいコンポーネントを追加する際に参照できるテンプレートまたは既存コンポーネントの慣例がある
- [ ] SSOTにないパターンが必要になったとき、SSOTを先に更新してから実装する判断フローが存在する
- [ ] 不要になったトークンを検出・削除する手段がある（grep等で未使用トークンを特定できる）
- [ ] トークンを削除する際に、移行先のトークンまたは代替値が明示されている

### 8. 壊れ方が設計されている

コンポーネントは想定外の入力やデータ欠損に対して、予測可能な方法で劣化（graceful degradation）すべきである。
変更は「拡張には開き、修正には閉じる」（開放閉鎖の原則）に従い、既存の利用箇所を壊さない形で行う。

- [ ] テキストコンテンツが想定より長い場合のふるまい（truncation / wrap / scroll）がコンポーネントごとに定義されている
- [ ] 必須データが欠損した場合のフォールバック表示（空状態・プレースホルダー）が定義されている
- [ ] コンポーネントが受け取るデータの境界条件（0件・1件・大量件数）での表示が考慮されている
- [ ] 想定外のプロパティの組み合わせに対して、サイレントに壊れるのではなく、安全なデフォルトにフォールバックする
- [ ] 変更が「安全な変更」（新バリアント追加・任意プロパティ追加）か「破壊的変更」（プロパティ名変更・選択肢削除・必須プロパティ追加）かを区別して判断している
- [ ] 不要になったコンポーネントには非推奨マーク（Deprecated）と代替先が明示され、削除トリガー（利用箇所ゼロ・次メジャーバージョン等）が設定されている

### 9. コンポーネントの状態が網羅されている

コンポーネントが取りうるすべての視覚的状態が列挙され、デザイン・実装の両方で確認されている。
UIスタック（Scott Hurff）の5状態を基準とする: 理想状態・空状態・読み込み中・不完全状態・エラー状態。

- [ ] 各コンポーネントの状態一覧（例: デフォルト・操作中・無効・読み込み中・エラー・空）が定義されている
- [ ] 不完全状態（一部データのみ存在する場合）のフォールバックルールがデータ項目ごとに定義されている（例: 画像欠損→イニシャルアイコン、スコア未評価→「—」表記）
- [ ] 状態の組み合わせ（例: 無効 × エラー）で矛盾する表示がないことが確認されている
- [ ] 状態間の遷移（例: 読み込み中 → 完了、読み込み中 → エラー）が網羅されている
- [ ] プロパティ（外部から渡す設定値）と状態（内部で管理する値）が明確に区別されている
- [ ] ホバーで開閉する dropdown / popover / submenu は、**トリガーと展開パネルの間に物理的な gap を作らない**（hover bridge パターン）。視覚的に余白を取りたい場合は内側 wrapper の `padding` で表現し、外側 wrapper の hover 判定領域内に含める。`margin` で離すと隙間にカーソルが入った瞬間に閉じる「隙間ホバー切れ」が発生する
- [ ] ホバー状態が「視覚的に見える」だけでなく「キーボード操作（focus/blur, Tab, Esc）」でも同等に再現可能。ChevronDown 等の状態インジケータがあれば、開閉に応じて `transition-transform`（rotate 等）で状態を可視化する

### 10. 経路依存性を認識し、妥協の基準がある

既存のコードベースやデザイン資産は過去の意思決定の蓄積である。理想状態への移行には段階的なアプローチが必要になる。
追加と削除のコストは非対称である（追加は数分、削除は数日〜数週間。削除コストは時間とともに急激に上昇する）。

- [ ] 既存の逸脱（deviation）がSSOTまたはauditレポートに記録されている
- [ ] 逸脱ごとに「即修正」「次サイクルで修正」「許容する（理由付き）」の判断がされている
- [ ] 一貫性を優先する領域（例: カラートークン）と個別最適化を許容する領域（例: 特殊なレイアウト）が区別されている
- [ ] 新規コードは原則としてIDEAL準拠とし、既存コードの逸脱を理由に新規コードで同じ逸脱を許容しない
- [ ] 意図的な妥協には理由が記録されている（「なぜ今こうするか」を一言残す。Figmaコメント・コードコメント等）
- [ ] 妥協は後から直しやすい方向に倒されている（例: String仮置き→Enum絞込みは安全、逆は破壊的）
- [ ] 「今はやらない」と「やらなくていい」が区別され、先送りには着手トリガーが設定されている

### 11. Aesthetic identity がコミットされている

一貫性だけでは「壊れていないだけのプロダクト」になる。SSOTには「どう揃えるか」だけでなく「どう尖らせるか」を記録する。
詳細は `AESTHETIC.md` を参照。

- [ ] SSOTに `aesthetic` セクション（または相当の領域）が存在し、tone が1つ明示されている（brutally minimal / maximalist / editorial / playful 等）
- [ ] differentiation（記憶に残る一手）が1つに絞られて記録されている
- [ ] anti-direction（このプロジェクトでやらないこと）がSSOTの `antipatterns` に明記されている
- [ ] generic AI aesthetics の典型パターンが antipatterns に含まれている
  - generic フォント直書き（Inter / Roboto / Arial / system-ui / Space Grotesk 等を SSOT のホワイトリスト外で使用）
  - 紫グラデーション on 白背景の安易な使用
  - 中央寄せヒーロー → 3カラム機能 → CTA という定型レイアウトを意図なく採用
  - 装飾なしの単色ベタ塗り背景にデフォルトする実装
- [ ] aesthetic direction と実装の強度が整合している（minimalist宣言 × 装飾過多実装 のような不一致がない）

### 12. Motion がトークン化されている

モーションは generic に倒れやすい次元の1つ（無意味な fade-in が全要素に降りかかる等）。
duration / easing / 用途を SSOT で管理し、aesthetic と整合させる。

- [ ] duration がトークン化されている（例: `motion-fast` / `motion-base` / `motion-deliberate`）
- [ ] easing がトークン化されている（少なくとも standard / emphasized / decelerate / accelerate 相当の役割が定義されている）
- [ ] アニメーションを使う「意味」が SSOT またはコンポーネント定義に記録されている（散発的な micro-interaction を全要素に振っていない）
- [ ] `prefers-reduced-motion` への対応方針が SSOT に書かれている（無効化 / 短縮 / 代替表現のいずれか）
- [ ] aesthetic direction と整合している（brutally minimal で過剰モーションがない / playful で動きがない、のような矛盾がない）

### 13. 装飾レイヤー（visual depth）が意図されている

solid color にデフォルトせず、必要な atmospheric depth を SSOT で管理する。
ただし aesthetic direction に矛盾する装飾（例: brutally minimal で grain overlay）を入れない。

- [ ] 背景の方針が SSOT に明示されている（plain / gradient mesh / noise / geometric pattern / layered transparency 等から選択）
- [ ] shadow がトークン化されている（用途別に少なくとも 2〜3 段階：subtle / standard / dramatic 等）
- [ ] border / divider のスタイルがトークン化され、装飾用と機能用が区別されている
- [ ] テクスチャ・パターン・装飾用画像を使う場合、トークンまたはアセット参照経由で管理されている（コンポーネントに散在していない）
- [ ] Typography の display フォントと body フォントが分離されている（同一フォントで太さだけ変える構成になっていない、または意図的にそうしていることが明記されている）
- [ ] 配色が支配色＋鋭いアクセント構造になっている（5色均等配分のような generic な配分になっていない）

## スタック別の検証基準

### Tailwind CSS v4

- [ ] `@theme` ブロックにReference tokensが定義されている
- [ ] `:root` / `.dark` 等でSystem tokensがCSS Custom Propertiesとして定義されている
- [ ] `text-[#xxx]` `w-[123px]` のようなJIT任意値が使われていない（例外はSSOTに明記）
- [ ] コンポーネントに `className` の直書きが散在していない（共通クラスはコンポーネントに集約）

### Tailwind CSS v3

- [ ] `tailwind.config` にカラー・スペーシング・フォントのカスタムトークンが定義されている
- [ ] `theme.extend` 外にデフォルト値の上書きがない（意図的な場合はコメントあり）
- [ ] `text-[#xxx]` `w-[123px]` のようなJIT任意値が使われていない（例外はSSOTに明記）
- [ ] コンポーネントに `className` の直書きが散在していない（共通クラスはコンポーネントに集約）

### CSS Modules

- [ ] 変数が `:root` またはテーマファイルで一元管理されている
- [ ] モジュール間で同じ値を別々にハードコードしていない

### Unity (C# / ScriptableObject)

- [ ] カラー・スペーシングがScriptableObjectまたはstatic classで一元管理されている
- [ ] Inspectorで直接色を設定しているオブジェクトがない（SSOTを参照している）
- [ ] UI PrefabがSSOTのトークンを参照している

### その他

- デザイントークンを定義するファイルが1つ存在し、他はそれを参照していれば準拠とみなす

## 評価の運用

各チェックは以下の3段階で評価する：

- **✅ 準拠** — 基準を満たしている
- **⚠️ 部分的** — 一部満たしていない、または例外が多い
- **❌ 非準拠** — 基準を満たしていない

`AUDIT.md` の実行時にこのチェックリストを使い、⚠️と❌の項目を修正対象とする。
