# Instruction file 肥大化への整理戦略

公式・著名エンジニアの推奨を踏まえた整理戦略。問題種別ごとに採用すべき戦略を示す。

公式の引用と URL は `.cursor/skills/instruction-refactor/references/official-criteria.md` を参照。

## 戦略 1: Ruthlessly prune（容赦なく削る）

> "Keep it concise. For each line, ask: 'Would removing this cause Claude to make mistakes?' If not, cut it."
>
> 出典: Codex best-practices

最も基本的な戦略。各行について「これを消したら Codex が間違うか？」を自問する。No なら削除。

適用する場面:

- 判定 1: 公式上限超過
- 判定 2a: 責務越境（実装詳細を削除）
- 判定 2b: SSOT 逸脱（抜粋を削除し参照のみに）
- 判定 2d: 二重説明（片方を削除）

## 戦略 2: Progressive Disclosure（段階的開示）

skill のオンデマンドロード設計を活用する:

1. セッション開始時: `name` + `description` のみがプリロードされる
2. Codex が関連性を判断時: `SKILL.md` 全文がロードされる
3. 必要に応じて: `SKILL.md` から参照される追加ファイルがロードされる

ファイル構造:

```text
my-skill/
├── SKILL.md                 # 本体（500 行以内）
├── references/<topic>.md    # 詳細リファレンス、オンデマンド
├── scripts/<task>.<ext>     # 決定論的処理（Codex が exec）
└── assets/<file>            # テンプレート・固定アセット
```

適用する場面:

- 判定 1: SKILL.md 500 行超過
- 判定 2c: DRY 違反（共通骨格を `references/` に集約 + 両方から参照）

実装ガイド:

- 共通骨格を別 skill 配下の `references/` に置き、別 skill から `.cursor/skills/<owner>/references/<file>.md` で参照する形は OK（公式が明示的に推奨はしていないが、パスベースの Read で動作する）
- SKILL.md 本体には「詳細プロトコル: `<path>` を参照」と書き、要点だけ残す

## 戦略 3: Import (`@path`)

> "If you have an `AGENTS.md` file, you can source it in your `CLAUDE.md` using `@AGENTS.md` to maintain a single source of truth."
>
> 出典: Anthropic Docs（Simon Willison が引用）

CLAUDE.md / AGENTS.md レベルでのモジュール分割に有効。

```markdown
See @README.md for project overview and @package.json for available npm commands.

# Additional Instructions
- Git workflow: @docs/git-instructions.md
- Personal overrides: @~/.cursor/my-project-instructions.md
```

適用する場面:

- CLAUDE.md が肥大化し始めたが完全に消すわけにはいかない情報がある
- 階層メモリ（user / project / local）の使い分け

## 戦略 4: 階層メモリの使い分け

「常時ロード不要だが場面では必要」な情報は `skills/` に切り出すのが推奨。

> "CLAUDE.md is loaded every session, so only include things that apply broadly. For domain knowledge or workflows that are only relevant sometimes, use skills instead."
>
> 出典: Codex best-practices

`AGENTS.md (shared SSOT)` には以下のみを残す:

- 全プロジェクトで常時必要な振る舞い・規約
- ファイル横断のルール（命名・git・書式）
- エージェント・スキルの呼び出し方針

ドメイン特化の手順・テンプレート・チェックリストは skill 化する。

## 戦略 5: 棚卸し（dead code 削除）

> Armin Ronacher (Flask 作者): "Agentic Coding Things That Didn't Work" (2025-07-30)
> 多数作成したスラッシュコマンドのほとんどを未使用化のため **削除した** と報告

定期的に「使われていない」コマンド・スキル・観点を棚卸しして削除する。`pir_skill_log.md` / `pir_planner_log.md` などのログから「最近 N 回起動されていない skill」「触れられていない agent 観点」を抽出する。

適用する場面:

- 全体棚卸しの定期メンテ（数ヶ月に 1 回）
- skill 数が増えすぎて管理コストが目立ち始めたとき

### dead code 削除後の残存ゼロ確認（同一ファイル内全セクション grep）

ファイル内の dead code 削除 Edit を適用したら、削除したキーワードを**同一ファイル全体に grep で残存確認**し、残存ゼロを機械確認して初めて「削除完了」とする。

理由: 入力セクションを削除しても、**返り値テンプレ / プロセス手順 / 出力フォーマット内の参照フィールド**に残骸が残るケースが繰り返し発生する（dead code は同一ファイル内の複数セクションに分散しているため、最初の検出で全箇所を拾いきれない）。最後に `grep -n <キーワード> <file>` で一掃確認すれば、複数イテレーションかかるところが 1 回で完結する。

## 戦略 6: 意味的重複の統合（正規化）

単一ドキュメントを加筆し続けた結果、同じ概念・ルール・手順が複数箇所に分散したときの正規化。**要約・圧縮（情報を削って短くする）ではなく、重複を 1 箇所に集約して情報量を保つ lossless な効率化**。

手順:

1. **クラスタリング** — ファイルを通読し、意味的に重複する段落 / ルールをクラスタにまとめる（判定 2d）
2. **統合先の決定** — 各クラスタの集約先を 1 箇所決める（最初の出現箇所、または最も論理的に属するセクション）
3. **和集合での統合** — クラスタ内の各箇所が持つ固有情報を **すべて** 統合先に取り込む。差分がなければ単純に 2 つ目以降を削除、差分があれば和集合を取る
4. **残余の処理** — 統合先以外の箇所は削除する（同一ファイル内で参照が必要なら `→ §<見出し> を参照` の内部参照に置換）
5. **情報点包含チェック（lossless 検証）** — 以下の決定論的手順で照合する。モデルの記憶頼みにしない:
   1. **統合前**に、クラスタ内の各箇所から atomic な assertion（情報点）を **1 行 1 点**で箇条書き抽出する（「誰が・何を・どの条件で」を 1 文に割る）
   2. 抽出した全 assertion に通し番号を振り、`[ ]` のチェックリストにする
   3. **統合後**の文面を読み、各 assertion が（言い換えを許容して）含まれているかを 1 点ずつ照合し `[x]` を付ける
   4. 全点 `[x]` になって初めて統合確定。未チェックが 1 点でも残れば、その点を統合先に追記してから再照合する

   これは「圧縮（情報を削って短くする）になっていない」ことの機械的担保。作ったチェックリストは手順 6 の diff 提示に添える
6. **diff 提示** — 確定前に変更前後の diff をユーザーに見せて承認を得る

適用する場面:

- 判定 2d: 同一ファイル内の二重説明 / 意味的重複
- 単一ドキュメントの加筆運用で蓄積した重複の棚卸し

> 圧縮との違い: 「言い換えて短くする」のは情報が減るため **やらない**。あくまで「重複している 2 つ目以降を消す / 1 箇所にまとめる」だけ。情報量は保存される。

## 性能保全ゲート（外出し・prune を実行する前に必ず通す）

リファクタで最優先するのは「肥大化の解消」ではなく **指示の実効性（= Claude / subagent が実際に正しく動く性能）を落とさないこと**。DRY 違反・SSOT 逸脱・公式上限超過の解消が、情報の**配達経路**を壊して性能を下げることがある。外出し（戦略 2）と prune（戦略 1）を実行する前に、各候補をこのゲートに通す。

### なぜ外出し・prune が性能を下げうるか

- **subagent は Task subagentを呼べず、単独・並列で起動される**。親（オーケストレーター）が起動時に読んだ参照を引き継がない。運用上クリティカルな指示（実行するクエリ・コマンド、出力フォーマット定義、必ず守るチェック手順）を別ファイルに外出しすると、その subagent が参照を読む保証がなく、読み漏らせば指示が欠落して性能が落ちる
- **inline のコピーが唯一の配達経路**であることがある。SSOT 逸脱（= ある SSOT の内容を inline 複写）に見えても、その内容を消費する agent が当該 SSOT を**実際に Read する手順を持っていない**なら、inline コピーこそが配達経路。prune すると消費側に情報が届かなくなる
- 「親 SKILL は起動時にこの共通規約を Read する」と書いてあっても、それは**親が読むだけ**で、その親が起動する subagent も読むとは限らない。subagent 側の定義に Read 手順が無いなら配達経路は別

### ゲート判定（各候補について順に問う）

1. **その内容は「常時ロードされる単一ファイル内」で完結する重複か？**（同一 CLAUDE.md 内 / 同一 SKILL.md 内 / 同一 agent 定義内の二重説明）
   - YES → 外出し不要。戦略 6（同一ファイル内で和集合統合）で安全に解消できる。**性能影響なし**。実施してよい
2. **prune 対象は「inline の SSOT 複写」か？ なら消費側 agent はその SSOT を実際に Read する手順を持つか？**
   - 消費側が SSOT を読む（起動手順・Phase 手順に Read 指示が grep で確認できる）→ prune して参照化してよい
   - 消費側が SSOT を読まない（inline コピーが唯一の配達経路）→ **prune しない**。inline を残す（重複は配達の対価として許容）
3. **外出し対象は subagent が「実行する運用指示」か（クエリ・コマンド・出力テンプレ・必須チェック手順）？**
   - YES → 原則 inline 維持。どうしても共通化するなら、外出し先を作るだけでなく **各 subagent の定義に「Phase N で `<path>` を Read してから実行」という必須 Read 手順を明記**し、Read 1 回分のコスト増を受け入れる。手順を明記できない / 並列起動で読み漏らしが怖いなら外出ししない
4. **メイン（常時ロード文脈）の「気づき」だけが目的の inline か？**（実行はしないが把握しておくと良い情報）
   - 1 行ポインタ + 要点 1 行に圧縮してよい（実体は SSOT 側に残す）。ゼロにはせず「何が SSOT 側にあるか」の手がかりは残す

### 原則のまとめ

- **安全（実施してよい）**: 単一ファイル内の二重説明統合（戦略 6）／ dead 情報の削除（戦略 5）／ 消費側が確実に Read する SSOT への prune（戦略 1）
- **危険（既定で見送り）**: subagent の運用指示（クエリ・出力テンプレ・必須チェック）を別ファイルへ外出し／消費側が読まない SSOT 複写の prune
- 危険側を実施する場合は **必須 Read 手順の明記** をセットにし、それができないなら inline 維持を選ぶ
- **「行数を最も減らせる候補」が「最も性能を下げる候補」と一致しやすい**点に注意（大きな inline ブロックほど配達経路として重い）。削減量ではなく**配達経路の保全**で採否を決める

## 戦略選択フローチャート

> ℹ️ **この対応表が「問題種別 → 整理戦略」の SSOT**。SKILL.md ステップ 6 と checklist.md はここを参照する。齟齬が出たら本フローチャートを正とする。

0. **（全候補共通の前段）性能保全ゲートを通す** — 上記ゲートで「危険」に分類された候補は、必須 Read 手順の明記とセットにできない限り見送る
1. **公式上限超過？**（SKILL.md > 500 行 等）
   - YES → 戦略 2 (Progressive Disclosure)
2. **DRY 違反？**（複数ファイルでセクションが字句同一）
   - YES → 戦略 2 (共通骨格を references に外出し)
3. **SSOT 逸脱 / 責務越境？**
   - YES → 戦略 1 (Ruthlessly prune)
4. **二重説明 / 同一ファイル内の意味的重複？**（加筆で同じことが複数箇所に分散）
   - YES → 戦略 6 (意味的重複の統合。和集合で 1 箇所に集約 + lossless 検証)
5. **CLAUDE.md / AGENTS.md レベルの分割？**
   - YES → 戦略 3 (Import) + 戦略 4 (階層メモリ)
6. **全体的に膨らんでいるが具体的な悪さが見えない？**
   - YES → 戦略 5 (棚卸し)
7. **description 不適切？**（pushy パターン未準拠 / トリガー語句の過不足）
   - YES → `/skill-creator` の Description Optimization 実行を提案
8. **プロジェクト固有名の混入？**（ユーザースコープに具体名）
   - YES → プロジェクトスコープに移すか、本文ではなく参照に変える

## アンチパターン

- **「将来の拡張性」だけを理由に references/ を切る**: 現時点で具体的に分割が必要という根拠がなければ、既存パターンに合わせて単一ファイル構成で始める
- **SSOT を切らずに references を増やす**: 共通骨格を出した先がさらに別の SSOT と重複していたら根本解決にならない
- **削減と称して必要な指示まで消す**: "Would removing this cause Claude to make mistakes?" の自問を必ず行う
- **subagent の運用指示を「DRY だから」だけで外出しする**: 単独・並列起動の subagent は外出し先を読み漏らす。性能保全ゲートの判定 3 を必ず通す（行数削減より配達経路を優先）
- **「親 SKILL が読む共通規約」に subagent 固有の指示を集約する**: 親が読んでも subagent には届かない。subagent 側に Read 手順が無いなら inline 維持
- **削減後に動作確認をしない**: 大規模リファクタは `/pir2` 経由でレビューを通す

## 大規模リファクタは `/pir2` 経由

5 ファイル以上に影響する整理を行う場合、独断せず `/pir2` で planner → reviewer のレビューを通すことを推奨。`/instruction-refactor` 自体は **検出と小〜中規模のリファクタ実行** までを担うが、複雑な構造変更は PIR² の品質保証を経由する設計。
